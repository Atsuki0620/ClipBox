## 0. この文書の歩き方（読み方ガイド）

この文書は「ClipBox のランキングが体感的に劣化している」という問題に対して、**次にどの改善を実装すべきか**を
データで選べるようにするための分析レポートである。実装はまだ行わない。読者として **(1) レビューする人間** と
**(2) 続きを引き継ぐ将来の AI エージェント** の両方を想定し、前提知識ゼロでも追えるよう用語と手法を本書内で
完結させた。

**この版で一番大事な一文**（§12 の要約）:
> **「順位がどれだけ動くか（効果）」は安全性ではない。** 順位が動くこと自体は、汚染を直しているなら善・
> genuine を巻き込んでいるなら悪で、その符号を決めるのは **精度**（genuine を誤って捨てていないか）である。
> ゆえに **安全（精度・可逆性）と体感改善（効果）は相反せず、直交する**。

**用語の最重要 3 つ**（ここだけ先に覚えると以降が読める）:
`view_days`（視聴日数。順位の主変数）/ `composite`（本番既定のスコア式）/
`FILE_ACCESS_DETECTED`（過去にだけ書かれた「ファイルアクセス検知」由来の履歴。汚染の主因）。

## 1. エグゼクティブサマリー

- ランキング上位は、ユーザーがアプリで**実際に再生した日数**ではなく、過去にだけ記録された
  **旧ファイルアクセス検知（`FILE_ACCESS_DETECTED`）の日数**で大きく決まっている。これが
  「いつも同じ動画が上位」「期間を変えても順位が変わらない」という体感劣化の中心的な原因と考えられる。
  **つまり現状のランキングは『正しい状態』ではない。**
- この FILE_ACCESS 行は **1 件ずつのループで書かれ、`viewed_at` に視聴時刻ではなくファイルの `st_atime`（OS 最終
  アクセス時刻）を入れていた**ことを archived コードで確認（→ §5.1）。実測でも FILE 行は**少数の日に集中**
  （59 日に 5,443 行・上位 10 日で 54.2%）し、少数のバルクイベントが多数の「視聴日」に水増しされている（→ §6.4）。
  なお「atime ＝ コピー時刻」までは現在のデータで断定できない（V1 同日一致 2.0%）ため、本書は断定しない。
- 履歴 6,725 行のうち **5,443 行（80.9%）が `FILE_ACCESS_DETECTED`**、現行アプリが書く `APP_PLAYBACK` は
  1,282 行（19.1%）のみ。`view_days` はこの偏った母集団の上に成り立っている。
- **「順位が動かない＝安全」ではない。** 現状ランキングが汚染で歪んでいる以上、順位を温存することは
  「壊れた状態を保存する」ことに等しい。本来の安全は、**genuine（実再生・正当な視聴）を誤って捨てないこと（＝精度）**と
  **戻せること（＝可逆性）**であって、順位の動かなさではない（→ §12）。
- 高精度に「明白なバルク」と断定できるのは **C1（完全一致タイムスタンプ ≥5 件の旧 FILE_ACCESS）= 63 行 / 61 動画**
  だけ。だが C1 を隔離してもランキングはほぼ動かない（Top50 入替 0、Spearman 0.996）。これは「C1 が安全だから良い」
  のではなく、**C1 では壊れた順位をほとんど直せない＝正しいが不十分**という意味である。**C1 は体感改善策ではなく
  衛生（hygiene）＋保護テストの土台**。
- 体感を動かす候補（FILE 全除外 C4/C5、60 秒クラスター C7b）が危険なのは「順位が大きく動くから」ではない。
  **genuine な過去視聴まで巻き込む（＝精度が低い）から**である。順位が大きく動くこと自体は、汚染を直す
  この問題ではむしろ望ましい。危険の本体は「動く量」ではなく「誤って捨てる量」。
- **「効きつつ壊しにくい」現実解は C6（FILE 日の重み下げ）**。除外せず減点するだけで Top50 を 7〜14 件動かし、
  Spearman 0.75〜0.86 を保ち、係数を変えれば**可逆**。「この行はニセ」と断定しないので精度リスクが小さい。
- もう一つの体感劣化要因は**外付け HDD の不可視化（C8）**。ライブラリ 4,103 本のうち 3,782 本（**92%**）が
  `is_available=0` でランキングに出ず、可視ランキングは実質「C ドライブ内ランキング」になっている。汚染対策とは
  独立した、最大の体感ポテンシャルを持つ別軸。

| 段階 | 内容 | 理由 |

## 2. 背景と問題意識

### 2.1 観測された体感劣化

ClipBox のランキング画面（`/api/ranking`）で、ユーザーは次の不満を感じている。

- **「いつも同じ動画が上位に並ぶ」** — 上位の顔ぶれが固定化していて新鮮味がない。
- **「期間（今週/今月/全期間）を変えても順位がほとんど変わらない」** — 期間フィルタが効いている実感が薄い。

ランキングの主変数は `view_days = COUNT(DISTINCT DATE(viewed_at))`（視聴した「日」の数）である。もし
`viewed_at` の大半が短期間に一括投入された人工的なレコードであれば、(a) 一部の動画だけが不自然に高い
`view_days` を持ち続け、(b) その大半が特定の過去日に固まっているため期間を絞っても順位が動かない、という
両方の症状が説明できる。**この場合、現状の順位は「正しい」ものではなく、修正後に順位が動くことこそ正常化である。**

### 2.2 汚染源 — `FILE_ACCESS_DETECTED` レガシー

ClipBox は過去に「ファイルへのアクセスを検知して視聴履歴に記録する」機能を持っていた（`viewing_method =
'FILE_ACCESS_DETECTED'`）。この機能は Phase 1 で廃止され、現在アプリが書き込む履歴は `APP_PLAYBACK`
（アプリ内再生）だけになっている。しかし**過去に書かれた `FILE_ACCESS_DETECTED` 行（5,443 行）は DB に残ったまま**で、
`view_days` の計算に今も算入されている。**この書き込み機構は archived コードで確認済み**で、ファイルアクセス検知は
**1 件ずつのループで INSERT し、`viewed_at` に `datetime.now()` ではなくファイルの `st_atime`（OS 最終アクセス時刻）を
入れていた**（→ §5.1・§6.4）。実測でも FILE 行は**少数の日に極端に集中**（59 日に 5,443 行・上位 10 日で 54.2%・
最大の 1 日で 12.9%）しており、`view_days` を実態以上に押し上げている。

ただし注意すべきは、`FILE_ACCESS_DETECTED` は一様な「ゴミ」ではない点である。**「ファイルを開いた痕跡」は
本来「視聴の代理指標」であり、ユーザーが実際に開いて見た genuine な記録も多く含む。** 純粋なスキャン副産物
（バルク）から genuine な利用までが連続的に混ざっている。だから「FILE は全部ニセ」という二値判定は精度を
大きく損なう（→ §12〜13）。

### 2.3 これまでの調査と本分析の問い

Round 1 / Round 2 の先行調査（`docs/analysis/ranking_bulk_viewing_history_investigation.md`・main マージ済み）で、
「完全一致タイムスタンプ ≥5 件かつ FILE_ACCESS かつ AVP 導入前」の行が**唯一の高精度なバルク汚染フラグ**
（11 グループ / 63 行 / 61 動画・`APP_PLAYBACK` 混入ゼロ）であると分かっていた。

本分析の問いは次の 2 つである。

1. 先行調査の数値を**独立に再現**できるか。
2. 先行調査で未定量だった候補（APP 限定ランキング・重み下げ・genuine_view_days・availability バイアス）まで
   含めて 8 候補を、**「精度（genuine を捨てないか）／可逆性（戻せるか）／効果（順位をどれだけ正常化するか）／
   根拠（仕様と整合するか）」の軸**で定量比較し、**次に実装する 1 つを選べる状態**にできるか。

## 3. 用語定義（本書内で自己完結）

| 用語 | 定義 |
|---|---|
| **viewing_history** | 視聴イベントのログテーブル。1 行 = 1 回の視聴記録（`video_id`, `viewed_at`, `viewing_method`）。 |
| **viewing_method** | 視聴記録の出所。後述 3 値。 |
| `APP_PLAYBACK` | 現行アプリでの再生（`play_video()` 単発、または AVP 一括）。**現在も書かれる唯一の方式**。genuine の確実な印。 |
| `FILE_ACCESS_DETECTED` | 旧「ファイルアクセス検知」由来。**現在は書かれない（archived）**。書き込みは 1 件ずつループ・`viewed_at=st_atime`（OS 最終アクセス時刻）。本分析の汚染対象だが、genuine な利用も混在し得る。 |
| `MANUAL_ENTRY` | 手動入力。本 DB には**0 行**（実質存在しない）。 |
| **view_count** | 動画ごとの視聴**行数** = `COUNT(*)`（同日複数回も別カウント）。 |
| **view_days** | 動画ごとの視聴**日数** = `COUNT(DISTINCT DATE(viewed_at))`。**順位の主変数**。ただし FILE 期は `viewed_at=st_atime` のため、FILE 由来の view_days は「熟視した日」ではなく**「OS がそのファイルにアクセス時刻を刻んだ日」の distinct 数**を数えている（→ §6.4）。 |
| **likes** | 動画ごとの「いいね」数（`likes` テーブルの行数）。本 DB では 67 動画 / 91 行のみと希少。 |
| **composite** | 本番ランキングの既定スコア式。§4 で詳述。`view_days`・`likes`・tier ボーナスの合成。 |
| **t1 / Tier1** | 「判定済み」フラグ。`current_favorite_level >= 0`（レベル付与済み）なら 1。composite に +50%。 |
| **t2 / Tier2** | 「選別済み」フラグ。`is_selection_completed = 1` なら 1。composite に +30%。 |
| **AVP** | Adjacent Video Player。最大 **4 本**を同時再生する機能。1 セッションで最大 4 行を**同一 `viewed_at`** で記録。 |
| **pre_avp 境界** | `2026-06-10 00:00:00`。commit `8bf84fb` で AVP の履歴記録（= `APP_PLAYBACK` 化）が始まった日時。これより前を pre-AVP、以後を post-AVP と呼ぶ。 |
| **genuine（正当視聴）** | 実際にユーザーが見た記録。`APP_PLAYBACK` は確実に genuine。FILE 行のうちにも genuine（実際に開いて見た）と非 genuine（スキャン副産物＝バルク）が混在する。 |
| **essential_filename** | 全プレフィックスを除いた本質ファイル名（DB の UNIQUE 識別子）。**本書には出さない**（private）。 |
| **is_available** | その動画が今ランキングに出る対象か。外付け HDD 未接続なら 0。本番ランキングは原則 `is_available=1` のみ表示。 |
| **storage_location** | 物理保管先（`C_DRIVE` / `EXTERNAL_HDD`）。 |
| **scope（スコープ）** | 集計の母集団。本番に存在するのは「利用可能のみ（`is_available=1`）/ すべて」の 2 つ。 |
| **lens（分析専用レンズ）** | 本書だけで使う追加母集団（`c_drive_only` / `external_hdd_only` / `all_not_deleted`）。**本番 UI には存在しない**。観察のために切る断面。 |
| **精度（precision）** | 候補が**落とす/減点する行のうち、本当に非 genuine（バルク）だった割合**。genuine を巻き込まないほど高精度。**本書で「安全」の中核**。 |
| **可逆性（reversibility）** | 適用後に**情報を失わず元へ戻せるか・段階調整できるか**。flag/重みは可逆、二値の全除外は段階調整不能で破壊的。本タスクは DB 物理削除を禁じるため、可逆性は「破壊的な価値判断をしないか／ダイヤルで戻せるか」を指す。 |
| **効果（体感改善 / impact）** | クリーニング適用で**順位がどれだけ動くか**（Top-N 入替・Spearman の低下）。**値中立**。汚染を直しているなら善、genuine を巻き込んでいるなら悪で、符号を決めるのは精度。安全性ではない。 |
| **Spearman 順位相関** | 2 つの順位列がどれだけ似ているかの指標（1.0=完全一致、0=無相関）。**本書では「効果（順位がどれだけ動いたか）」の指標であって安全性ではない。** 低い＝大きく動く＝体感効果が大きい。それが善か悪かは精度が決める。 |
| **Top-N 入替** | ベースライン Top-N のうち、候補適用後に Top-N から**外れた**動画の数。大きいほど顔ぶれが変わる＝効果が大きい（同じく値中立）。 |

## 4. 現行ランキングの仕組み

ランキングは `core/analysis_service.py::get_ranked_videos_for_tab(ranking_type, period_label, min_level,
availability_filter, top_n)` が生成し、`GET /api/ranking` で配信される。

### 4.1 4 つの種別（ranking_type）

| 種別 | スコア | 意味 |
|---|---|---|
| `view_count` | `COUNT(*)` | 視聴回数（行数） |
| `view_days` | `COUNT(DISTINCT DATE(viewed_at))` | 視聴日数 |
| `likes` | いいね数 | 純粋ないいねランキング |
| `composite`（既定） | 下記の合成式 | 視聴日数・いいね・判定/選別状態の総合 |

### 4.2 composite 式（本書の中心）

```
composite = int( (view_days · 1 + likes · 3) · (1 + 0.5·t1 + 0.3·t2) · 100 )
```

項ごとの意味:

- `view_days · 1`（係数 `_COMPOSITE_A = 1`）— 視聴日数の寄与。**ここに FILE_ACCESS 汚染が直接効く**。
- `likes · 3`（係数 `_COMPOSITE_B = 3`）— いいね 1 件は視聴日数 3 日分の価値。ただし likes は希少（67 動画のみ）。
- `(1 + 0.5·t1 + 0.3·t2)` — 判定済み（+50%）・選別済み（+30%）の**乗算ボーナス**。両方なら ×1.8。
- `· 100` — 整数化のためのスケール。

**タイブレーカー**（同点時の並び）: `score DESC → last_viewed_at DESC → id ASC`。
**フィルタ**: `score > 0` かつ `is_deleted = 0`（スコア 0 の動画はランキングに載らない）。

### 4.3 数式ワークスルー（匿名 1 本 = V001）

ベースライン Top1 の V001 を実データで分解すると（表3 参照）:

- アプリ再生日数 `app_view_days = 7`、旧ファイルアクセス日数 `file_access_view_days = 23`、`likes = 2`、`t1 = 1`、`t2 = 1`。
- 総 `view_days`（method を問わない distinct date）は、composite から逆算して **24 日**。
  `(view_days + likes·3) · 1.8 · 100 = 5400` ⟹ `(view_days + 6) · 180 = 5400` ⟹ `view_days = 24`。
- 検算: `int((24·1 + 2·3) · (1 + 0.5·1 + 0.3·1) · 100) = int(30 · 1.8 · 100) = 5400` ✓。

ここで重要なのは、**総 view_days 24 日のうち 23 日が「旧ファイルアクセスのあった日」**である点（app は 7 日、
同日に両方ある重複が 6 日 ⇒ 7 + 23 − 24 = 6）。つまり V001 の Top1 は、アプリでの実再生(7 日)ではなく
ファイルアクセス検知(23 日)が事実上決めている。これが「上位が実再生で決まっていない」という体感の正体を、
1 本で具体化した例である。**ただし file 由来 23 日のすべてがバルクとは限らない**（genuine な利用も混在し得る）
点が、後の精度議論（§12〜13）の核心になる。

## 5. 前提と制約

### 5.1 コードと Git 履歴から確認した事実

- 現在 viewing_history に書かれるのは `APP_PLAYBACK` のみ。`play_video()`（単発 INSERT）と
  `record_avp_viewing()`（AVP 一括 INSERT）の 2 経路で、いずれも `viewing_method = 'APP_PLAYBACK'`。
- `FILE_ACCESS_DETECTED` / `MANUAL_ENTRY` は廃止済み（既存行は保持）。後者は 0 行。
- **FILE_ACCESS_DETECTED の書き込み機構（archived コードで確認）**: `record_file_access_as_viewing`
  （`archive/video_manager_methods.py:153-162`）は `for file_info in accessed_files: INSERT ...` の**1 件ずつループ**で、
  各行の `viewed_at` に **`datetime.now()` ではなくファイルの `st_atime`**（`access_time =
  datetime.fromtimestamp(file_path.stat().st_atime)`、`archive/動画管理システム_技術仕様書.md:362`）を入れる。
  ⇒ FILE 行の `viewed_at` は**書いた時刻ではなく OS 最終アクセス時刻**で、行数はループなのでいくらでも出る。
- **Windows の atime 注意（過去のコードレビュー A-8）**: `NtfsDisableLastAccessUpdate` が既定有効のため atime は
  実アクセスで更新されないことが多く、しばしば**ファイルのコピー/作成時刻**を指す（`docs/reports/CODE_REVIEW_20260224.md:66-70`）。
  ※ ただし本 DB の `file_created_at` は**スキャンの度に上書き**され、ClipBox はレベル付与時にファイルをリネームするため、
  「atime ＝ コピー時刻」を**現在のカラムからは直接実証できない**（V1 で同日一致 2.0%・→ §6.4）。本書は**断定しない**。
- **完全一致 ≥5 が「バルク」である正しい根拠**（旧版の「単発1行/AVP4行だから生成不能」は FILE には不適切）:
  `viewed_at=st_atime` ゆえ、**5 本以上のファイルが秒一致の atime を持つ**のは、人が個別に視聴したのではなく
  1 回の機械的なファイル操作が同時刻を刻んだ時だけ。実測でも 11 群中 **7 群は複数フォルダ/ストレージに跨る**のに
  秒一致 timestamp を共有しており（→ §6.4 V3）、個別視聴では生成不能。
- **AVP は最大 4 本同時再生**。`record_avp_viewing()` は同一 `viewed_at`（`datetime.now()` 1 回分）で**最大 4 行**を
  `executemany` で一括 INSERT。これは **post-AVP / APP 側**の話で、`exact 2〜4` に正当 AVP が混ざり得るため、
  C1 は `FILE ∧ pre-AVP` ガードで AVP を分離する（このガードにより C1 の app_rows = 0）。
- `pre_avp 境界 = 2026-06-10 00:00:00`（commit `8bf84fb`）。これ以後の APP 完全一致グループは AVP 由来（最大 4 件）。
- 本番の availability スコープは「利用可能のみ / すべて」の 2 つだけ。本書の `c_drive_only` /
  `external_hdd_only` / `all_not_deleted` は**分析専用レンズ**であり、本番 UI には存在しない。

| 検証 | 対象 | 指標 | 結果 |
|---|---|---|---|

| **V4** | FILE 行の日付集中 | distinct 日数 / 上位1日 / 上位5日 / 上位10日 | **59 日 / 12.9% / 36.2% / 54.2%** |
| V4′（対照） | APP 行の日付集中 | distinct 日数 / 上位1日 / 上位10日 | 64 日 / 5.1% / 37.5% |
| **V2** | FILE viewed_at の時刻粒度 | 秒=00 率 / 真夜中率 / distinct H:M:S | 1.4% / 0.0% / 1,744 種 |
| **V3** | C1 完全一致≥5 群（11 群）の凝集 | 単一フォルダ群 / 単一ストレージ群 | 4 / 4（残り 7 群は**複数跨ぎ**） |

**この検証が示すこと（正直な切り分け）**:

- **確定（コード）**: FILE 行は 1 件ずつループで書かれ、`viewed_at = st_atime`。これは仮説ではなく archived コードの事実。
- **不成立（データ）**: 「atime ＝ コピー/作成時刻」という**直接の実証は得られなかった**。FILE `viewed_at` は
  現在の `file_created_at` と同日一致 2.0%・中央値 215 日ずれ。理由は §5.1 の通り `file_created_at` がスキャンで上書き＋
  リネームで原コピー時刻を失っているため。**よって本書は「コピー時刻説」を断定しない**（A-8 は背景仮説として残すに留める）。
- **強く支持（データ）**: ただし **FILE の view_days が「熟視」を表していない**ことは独立に裏付けられた。
  (i) **V4**: 5,443 行が 59 日に集中し、上位 10 日で 54.2%・1 日に 12.9%（≒700 行）。人が個別に見た記録なら 1 日に
  数百本の別動画が立つことはない＝**バルクなイベントの指紋**。APP（上位10日 37.5%・1 日 5.1%）と明確に対照的。
  (ii) **V2**: FILE の時刻は秒単位の実 OS 値（1,744 種の H:M:S・真夜中 0%）で、合成日付ではない＝atime 由来と整合。
  (iii) **V3**: C1 群は秒一致 timestamp を共有しつつ 7/11 群が複数フォルダ/ストレージに跨る＝個別視聴では生成不能な機械書き込み。
- **結論**: C1 の妥当性は「秒一致 timestamp ＝ 機械書き込みの指紋」（機構＋V2/V3）で揺るがない。そして**より大きな含意**は、
  FILE の view_days が**少数のバルクイベントを多数の「視聴日」に水増ししている**こと＝FILE をクラスとして弱シグナル化/
  除外する（C6/C4）根拠が強まり、行単位でバルクを狩る C3 の意義は薄れる（→ §13・§15・§16）。推奨（§17）は不変。

## 8. 分析方法（Methodology）

### 8.1 全体方針

実 DB に触れずに**本番ランキングを再現**するため、本番ロジック（`get_ranked_videos_for_tab`）を呼ばず
**SQL とスコア式をコピー DB 上で複製**した。スクリプトは `docs/analysis/notebooks/ranking_fairness_multifactor.py`
（依存は pandas / numpy のみ。scipy 非依存・Spearman は自前実装）。集計は CSV / `summary.json` に出力（すべて gitignore）。

### 8.2 動画ごとの集計（per-video）

各 `video_id` について以下を算出（期間は UI 既定の全期間）:

- `view_count = COUNT(*)`、`view_days = COUNT(DISTINCT DATE(viewed_at))`
- 内訳: `app_view_days`（APP 行だけの distinct date）、`file_view_days`（FILE 行だけ）、`c1_view_days`（C1 該当行だけ）
- `likes`、`t1 = (current_favorite_level >= 0)`、`t2 = is_selection_completed`
- `composite = int((view_days·1 + likes·3)·(1 + 0.5·t1 + 0.3·t2)·100)`、`last_viewed_at = MAX(viewed_at)`

### 8.3 候補フラグの算出ロジック

`pre_avp = viewed_at < '2026-06-10 00:00:00'`。`exact_size = ` 同一 `viewed_at` を持つ行数。

- **C1**: `exact_size >= 5 ∧ FILE_ACCESS ∧ pre_avp`
- **C2**: `exact_size ∈ [2,4] ∧ FILE_ACCESS ∧ pre_avp`
- **C3（短時間クラスター）**: FILE ∧ pre_avp の行を時刻順に並べ、**隣接ギャップ ≤ 60 秒**で同一クラスターに束ね、
  クラスターサイズ `≥5`（C3_5）/ `≥10`（C3_10）の行をフラグ。実装は `gap = dt.diff()`、`(gap > 60).cumsum()` で
  クラスター ID を振り、各 ID のサイズで閾値判定。
- **C4 / C5**: APP_PLAYBACK のみで再集計（= FILE 全除外。本 DB は MANUAL=0 なので両者同値）。
- **C6（重み下げ・除外しない）**: 日重み = **その日の行の method 最大重み**。`APP = 1.0` / `C1 = 0` /
  `C3_10 = 0.25` / その他 FILE = `file_base`（0.25 または 0.50）。`weighted_view_days = Σ 日重み` で composite 再計算。
- **C7（genuine_view_days）**: bulk 行を除いた distinct-date。`C7a` = C1 のみ除外、`C7b` = C1 + C3_10 除外。
- **C8（availability バイアス）**: 式は現行のまま、4 スコープで Top50 構成差を比較（除外でなくスコープ設計の問題）。

### 8.4 指標の定義

- **Top-N 入替**: ベースライン Top-N の動画 id 集合のうち、候補適用後に Top-N に**残らなかった**数。
- **Spearman**: ベースライン Top-N の各動画について（ベースライン順位）と（候補での順位）を並べ、両列を順位化して
  Pearson 相関を取る自前実装。候補で圏外に落ちた動画は `len(cand)+1` の最下位として扱う。
- これを **scope（4）× ranking_type（4）** の各組で算出 → 表2。headline は `all_not_deleted × composite`。
- **注意**: Top-N 入替・Spearman は**効果（順位がどれだけ動いたか）の指標**であって安全性ではない（→ §12）。

### 8.5 匿名化

ベースライン composite（all_not_deleted 順）に沿って実 `video_id` を連番 `V001…` に置換。対応表は
`docs/analysis/private/anon_id_map.csv` にローカル保存。公開する本書は匿名 ID と整数集計のみ。

## 11. 考察（仮説検証）

各仮説を §10 の数値で判定する。

- **H1（view_days は FILE_ACCESS 偏重）: 支持。** 表3 で Top50 全行が `file_vd ≫ app_vd`。V001 のワークスルー
  （§4.3）でも総 view_days 24 日中 23 日が file 由来。**上位は実再生でなく旧ファイルアクセスで決まっている**。
  これが「いつも同じ動画」「期間で動かない」体感の中心原因。**＝現状ランキングは正しい状態ではない。**

- **H2（exact ≥5 は高精度に隔離可能）: 支持。** C1 は 63 行すべて FILE ∧ pre-AVP、app_rows = 0。完全一致 ≥5 の
  グループ size 分布は {5:7, 6:1, 7:2, 8:1}（11 グループ）。`viewed_at=st_atime` ゆえ、**5 本以上が秒一致 atime を
  持つのは機械的なファイル操作の指紋**で、11 群中 7 群が複数フォルダ/ストレージ跨ぎ（§6.4 V3）＝個別視聴では生成不能。
  **ただし C1 適用の Top50 入替は 0**（susp_vd が上位にほぼ無い）。これは「安全だから動かない」のではなく、
  **明白バルクが上位の汚染のごく一部しか占めていない**＝C1 は精度満点だが効果は不足、という意味（→ §12）。

- **H3（FILE ∧ pre-AVP ガードが必須）: 支持。** post-AVP（≥2026-06-10）の 131 行はすべて APP で、その完全一致
  グループは最大 size = 4（AVP 上限と一致）。もし生の `exact ≥ 2` だけで判定すると、この正当な AVP 24 行を
  誤検出してしまう（精度低下）。`FILE ∧ pre-AVP` ガードを掛けることで、C1 の app_rows = 0 を達成できている。

- **H4（likes 不変）: 支持。** 表2 で likes ランキングは全候補・全スコープで Spearman = 1.000。いいねは
  viewing_history と独立なので、履歴クリーニングの影響を受けない。**「いいね」種別は今回の問題の埒外**。

- **H5（composite は view_days より頑健）: 支持。** 同一候補で比較すると、C4/C5 の external_hdd は composite 0.607 に対し
  生 view_days 0.229、C7b も composite 0.618 に対し view_days 0.298。`likes·3` と tier ボーナスが view_days の変動を
  緩衝している。**体感改善を狙うなら、生 view_days を直接いじるより composite ベースで段階導入する方が、
  誤爆の振れ幅を抑えられる（精度面で有利）**。

- **H6（availability の不可視化が主因の一つ）: 支持。** 表2 で `available_only` と `c_drive_only` の数値が
  ほぼ一致（例 C6_025 composite: available 12/16/0.753 vs c_drive 12/16/0.753）。母数も available_only 221 ≈
  C ドライブ寄り。ライブラリの 92% が外付け HDD で未接続時に消えるため、**可視ランキングは事実上 C ドライブ内の
  ランキング**になっている。これは FILE_ACCESS 汚染とは独立した、別軸の体感劣化要因。

### 総合的な解釈

体感劣化は単一原因ではなく、**(a) FILE_ACCESS 偏重の view_days（H1）** と **(b) is_available による外付け HDD
不可視化（H6）** の複合である。(a) の中で**高精度に「明白バルク」と断定できるのは C1 だけ**だが、それは上位を
作っている FILE 日数のごく一部（Top50 にほぼ寄与しない）でしかない。残りの大半は genuine とバルクが連続的に
混ざる**グレーゾーン**で、1 行ずつでは判別できない。したがって:

- **高精度・高可逆で確定できるのは C1 だけ**だが、それだけでは順位（＝体感）は正常化しない。
- **順位を正常化するには (a) はグレーゾーンを減点する C6（重み下げ）で段階的に、(b) は C8 スコープ設計で**
  取り組む必要がある。
- ここで旧版が「効果と安全性はトレードオフ」と書いたのは**誤り**だった。正しくは次節（§12）に置き換える。

## 12. 評価軸の再設計 ——「安全」を精度・可逆性・効果に分解する（本版の核心）

### 12.1 旧版の何が誤りだったか

旧版は候補を 1 本の `safety`（安全）スコアで評価していた。だがその数字には、**本来別物の 3 つ**が混ざっていた。

1. **精度** — 落とす/減点する行のうち、本当に非 genuine（バルク）だった割合。genuine を誤って捨てないか。
2. **可逆性** — 戻せるか・段階調整できるか（破壊的な価値判断をしないか）。
3. **順位の動かなさ** — クリーニング後にランキングがどれだけ保たれるか（高 Spearman・低 Top-N 入替）。

このうち **3 を「安全」に数え込んだのが誤り**である。なぜなら——

> **現状ランキングがバルク汚染で歪んでいる以上、「順位が変わらないこと」は良い状態の保存ではなく、
> 壊れた状態の保存である。** 順位の動かなさは安全性ではない。

具体的に、C1 の Spearman 0.996 を旧版は「C1 は安全」と読んだが、正しくは
**「C1 では壊れた順位をほとんど直せない＝正しいが不十分」**である。同じ数字が、前提（現状は正しい／現状は汚染）
次第で正反対の意味になる。安全の定義を「順位が動かない」に置くと、この**循環**（汚染した現状を基準に「動かない＝安全」と
評価し、汚染の修正を不安全と断じてしまう）に陥る。

### 12.2 「効果」は値中立であり、符号は精度が決める

3（順位の動き）は安全性ではなく、独立した **「効果（体感改善＝順位の正常化量）」** という値中立の軸として
切り出すべきものである。順位が動くこと自体に善悪はない:

- **精度が高いまま順位が動く** → 汚染だけを削って正常化している ＝ **善**。
- **精度が低いまま順位が動く** → genuine を巻き込んで歪めている ＝ **悪**。

つまり **効果の符号（良い変化か悪い変化か）は精度が決める**。「順位が大きく動く＝危険」という旧版の含意は誤りで、
**危険の本体は『動く量』ではなく『誤って捨てる量（＝低精度）』**である。

### 12.3 安全と体感改善は相反しない（直交する）

以上から、本来の安全 = **精度（genuine を捨てない）＋可逆性（戻せる）** であり、体感改善 = **効果（順位の正常化）**
である。この 2 つは**相反しない。直交する。** 旧版で相反して見えたのは、次の 2 点の人工物だった。

- **(i)** 効果の小ささ（順位の動かなさ）を「安全」に数え込んでいたため、定義上「安全 ↑ ⇔ 効果 ↓」が自動的に成立して
  いた（循環）。
- **(ii)** 高精度に隔離できる汚染（C1=63 行）がたまたま小さく、順位を動かす大きな塊（グレーゾーン）が genuine と
  分離しづらい。このため「効果の大きい候補ほど精度が低い」という**相関**が観測されたが、これは**検出の限界
  （epistemic）であって、効果と安全の間の必然法則ではない**。仮に精度の高い検出器で大量の汚染だけを除けたなら、
  それは「高効果かつ高精度」になり得る。

決定的なのは次の点である:

> **「行を消す」前提でのみ、効果↑＝genuine 巻き込み↑ のトレードオフが生じる。**
> 中間帯（グレーゾーン）は 1 行ずつでは artifact か genuine か判別できないので、消す量を増やすほど genuine を
> 巻き込む。だが **減点（C6）・再スコープ（C8）なら「この行はニセ」という per-row 判定を避けられる**ため、
> この per-row 分類問題を回避して順位を動かせる。トレードオフは「削除パラダイム」固有のもので、
> 重み付け/再スコープでは溶ける。

### 12.4 再設計後の評価マップ（精度 × 効果）

候補を「精度（縦・genuine を捨てないか）」×「効果（横・順位をどれだけ正常化するか）」で置くと:

| | 効果 小（順位ほぼ不動） | 効果 大（順位が大きく動く） |
|---|---|---|
| **精度 高（genuine を捨てない）** | **C1 / C7a**（安全だが効かない＝正しいが不十分） | **C6（重み下げ）/ C8（再スコープ）** ← 目標の象限（可逆でもある） |
| **精度 低（genuine を巻き込む）** | （該当なし） | **C4 / C5 / C7b**（効くが誤爆＝危険） |

- **C1 / C7a** = 高精度・高可逆・低効果。**衛生（hygiene）と回帰の安全網**。体感改善ではない。
- **C6 / C8** = 順位を実際に動かす（高効果）が、**減点・再スコープなので genuine を抹消せず可逆**。
  「効果あり」かつ「正しい意味で安全」＝**狙うべき象限**。
- **C4 / C5 / C7b** = 高効果だが、**genuine な FILE 期視聴を二値で全否定する（低精度）**ため危険。
  危険なのは Spearman が低いから（順位が動くから）ではなく、**誤って捨てるから**。

この再設計により、ユーザーの問い「安全と体感改善は相反するのか？」への答えは **「本質的には相反しない。
旧版の混同と検出限界の人工物だった。C6・C8 が安全（精度・可逆）を保ったまま体感を改善できる象限である」** となる。

## 13. 精度評価（genuine を巻き込まないか）

「落とす/減点する対象のうち、非 genuine（バルク）の割合はどれだけ高いか」。本書で**最重要の安全軸**。

| 観点 | 結論 |
|---|---|
| APP_PLAYBACK（確実な genuine）を巻き込むか | **全候補が巻き込まない**（対象の app_rows = 0）。完全一致 ≥5 グループ内の APP 行は**全 DB で 0 件**。ここは共通の床。 |
| AVP 正当 2〜4 件を巻き込むか | **C1 は巻き込まない**。post-AVP の APP 完全一致グループは**全て size = 4**（6 セッション・24 行）で AVP 上限と一致。完全一致 ≥5 に APP は 1 件も無い。生 `exact ≥ 2` だとこの 24 行を誤検出するが、`FILE ∧ pre-AVP` ガードで回避。 |
| genuine な FILE 視聴を巻き込むか | **ここで候補が分かれる。** C1=巻き込まない（明白バルクのみ・精度5）。C2=2〜4 件一致は弱く中程度（精度3）。C3/C4/C5/C7b=FILE 期の genuine 視聴を大量に巻き込む（精度1〜2）。C6=消さず減点なので「ニセ」断定を避ける（精度3）。 |
| グレーゾーンの扱い | genuine とバルクが連続的に混ざる FILE 行は 1 行ずつでは判別不能。**削除系（C3/C4/C5/C7b）はこれを丸ごと捨て精度を落とす。減点系（C6）は判別せず弱シグナル化するため per-row 誤爆を構造的に回避**。 |
| C1 が「明白バルク」である機構的裏付け | `viewed_at=st_atime`・1件ずつループ書き込み（archived コード）。秒一致 timestamp ×5本は機械書き込みの指紋で、§6.4 V3 で 11 群中 7 群が複数フォルダ跨ぎ＝個別視聴では不可能。**C1 の精度5 は機構＋実測で裏取り済み**。 |
| FILE をクラスとして弱める根拠（C6/C4 側） | §6.4 V4: FILE 5,443 行は 59 日に集中・上位10日で 54.2%＝**少数バルクの水増し**。V2: 秒単位の実 OS 時刻で合成日付ではない。⇒ FILE view_days は「熟視」を表さず、クラス減点（C6）の精度的妥当性が上がる。ただし「コピー時刻」断定はしない（V1 同日一致 2.0%・→ §6.4）。 |

**精度の順位**: C1 = C7a（最高） > C2 ≈ C6（中） > C7b ≈ C3_10（低） > C3_5 ≈ C4/C5（最低）。
C8 は汚染判定を行わない（genuine/バルクを区別しない）ため精度の概念が適用外（別軸）。

## 14. 可逆性評価（戻せるか・段階調整できるか）

本タスクは DB 物理削除・DELETE/UPDATE を禁じるため、**どの候補も DB レベルでは行を破壊しない**。したがって
可逆性は「**破壊的な価値判断をしないか／ダイヤルで段階的に戻せるか**」を指す。

| 候補 | 可逆性 | 理由 |
|---|:--:|---|
| C1（flag/reason） | ◎（5） | ランキングに触れない付加メタデータ。除外は別 PR に分離でき、外すだけで完全に元へ戻る。 |
| C6（重み下げ） | ◎（5） | 係数（0.25/0.5/…）を変えるだけの連続調整。行は残り、「ニセ」と断定しない。最も「効果がありつつ戻せる」。 |
| C8（スコープ設計） | ◎（5） | 表示母集団の設計選択。`last_known_score` 保存等で完全に切替・復帰できる。 |
| C2 / C3（監査フラグ） | ○（5※） | **監査フラグに留める限り**可逆。ただし自動除外へ昇格させると、低精度のまま二値判定になり実質非可逆化する。 |
| C4 / C5（FILE 全除外） | △（2） | 二値の全除外で段階調整できず、FILE 期の genuine を一括無価値化する破壊的判断。クエリ上は戻せても「弱めて様子を見る」ダイヤルが無い。さらに物理 DELETE に昇格させれば完全に非可逆。 |
| C7b（genuine 60 秒版） | △（2） | 実質 C3 の自動除外。C4/C5 と同様に二値・段階調整不能。 |

**読み取り**: 可逆性の観点でも C6/C8 が優れる。重み下げ・再スコープは**「効果を出しつつ、間違っていたら戻す/弱める」**
が両立する唯一のパターンであり、これが §12.3 の「削除パラダイムでのみトレードオフが生じる」の裏返しである。

## 15. 根拠の強さ評価

検出ルールが仕様・Git 履歴・コード挙動とどれだけ整合するか（精度とは別物。精度=結果の誤爆率、根拠=ルールの妥当性）。

| 観点 | 結論 |
|---|---|
| FILE_ACCESS 書き込み機構（archived コード） | ◎ `record_file_access_as_viewing`（`archive/video_manager_methods.py:153-162`）は **1 件ずつループ INSERT・`viewed_at=st_atime`**（`…技術仕様書.md:362`）。C1 の根拠は「書き込み口の行数上限」ではなく**この atime 機構**にある。 |
| 完全一致 ≥5 が機械書き込みである実測 | ◎ §6.4 V3: 11 群中 7 群が複数フォルダ/ストレージ跨ぎで秒一致 timestamp を共有＝個別視聴では生成不能。V2: 秒単位の実 OS 値（1,744 種 H:M:S）。 |
| FILE view_days が「熟視」でない実測 | ◎ §6.4 V4: 5,443 行が 59 日に集中・上位10日 54.2%（APP は 37.5%）＝少数バルクの水増し。 |
| 「atime ＝ コピー時刻」説 | △ **直接実証は得られず**（V1 同日一致 2.0%・file_created_at はスキャンで上書き＋リネームで原値喪失）。背景仮説（A-8）に留め**断定しない**。 |
| FILE_ACCESS の archived 扱いと整合 | ◎ FILE_ACCESS は現在書かれず、C1/C2/C3 対象は全て FILE_ACCESS かつ pre-AVP。 |
| 2026-06-10 境界・AVP 分離と整合 | ◎ post-AVP 行（131）は**全て APP**で完全一致は最大 size=4（AVP 上限）。C1 は `FILE ∧ pre-AVP` ガードで AVP と分離（app_rows=0）。 |
| 閾値・係数の妥当性 | C1 は機構＋実測に基づき恣意性が低い。C3（60 秒）・C6（重み 0.25/0.5）は**閾値/係数が恣意的**で、感度分析が必要。 |

**根拠の強さ順位**: C1 = C7a（最強：機構＋実測で裏取り） > C8（is_available 仕様は明確） > C4/C5（APP=現行有効方式は
妥当で、FILE が「熟視」でない実測が後押し。ただし genuine も含む全否定が弱点） > C2 ≈ C3 ≈ C6（閾値・重みが恣意的）。

## 16. 効果評価（体感改善＝順位がどれだけ動くか）

**値中立の軸**。大きく動くこと自体は、精度が高ければ「正常化」、低ければ「歪み」。符号は §13 の精度が決める。

| 候補 | 効果（順位の動き） | 精度との組み合わせ＝意味 |
|---|---|---|
| C1 / C7a | **ほぼ無し**（Top50 入替 0・Spearman 0.996） | 高精度 × 低効果 = **正しいが不十分**（衛生・土台） |
| C6（重み下げ） | **中**（Top50 7〜14・Spearman 0.75〜0.86） | 中精度 × 中効果 × 高可逆 = **効きつつ壊しにくい本命** |
| C4/C5（APP のみ） | **大**（Top50 22〜25） | 低精度 × 大効果 = **効くが genuine を誤爆（危険）**。比較ビュー向き |
| C7b（genuine 60 秒版） | **大**（Top50 21〜27・生 view_days で Spearman 0.3） | 低精度 × 大効果 = C4/C5 と同じく危険 |
| C8（availability） | **最大** | 別軸（汚染判定なし）× 大効果 × 高可逆 = **最大ポテンシャル**。外付け HDD 3,782 本（92%）の不可視化を是正 |

「いつも同じ動画」「期間を変えても順位が変わらない」体感の主因は、**(a) FILE_ACCESS 偏重の view_days** と
**(b) is_available による外付け HDD 不可視化** の複合。C1 単体では (a) のごく一部しか動かないため、
**体感改善の本命は C6（重み下げ）と C8（availability 設計）**。両者は **§12.4 の「精度高 × 効果大」象限**にあり、
安全（精度・可逆）を保ったまま順位を正常化できる。

## 17. 推奨する次の一手

### 初手（最初の小 PR）— 衛生であって体感改善ではない、と正直に位置づける

- **`EXACT_TIMESTAMP_GE5_LEGACY_FILE_ACCESS`（C1）の flag/reason 化のみ**。**ランキング除外はまだしない。**
- 派生テーブル/ビュー、または集計時フラグで `bulk_reason = 'EXACT_TIMESTAMP_GE5_LEGACY_FILE_ACCESS'` を付与。
- **APP/AVP 保護テストを追加**（完全一致 ≥5 に APP 行が無いこと／AVP 完全一致は size ≤ 4 であることを回帰で固定）。
- **位置づけ**: 精度・可逆性・根拠が最強の**衛生作業＋回帰の安全網**。**体感（順位）はほぼ動かさない**。
  これを「体感改善策」として売らない——順位を動かすのは次段（C6/C8）の役割。
- **なぜ初手にこれを選ぶか**: 唯一「app_rows = 0・誤爆ゼロ」を証明できる確定部分集合で、後続の C6/C8 が順位を
  動かしたとき「何が genuine で何がバルクか」の基準線（C1 隔離済みの状態）を与えるから。先に C6/C8 から入っても
  良いが、C1 を先に固めると効果の切り分けが容易になる。

### 本命の分析タスク（体感改善はこちら。安全＝精度・可逆を保ったまま順位を動かす）

1. **C6 重み下げの感度分析**（FILE day = 0.25/0.5、C1 day = 0、C3 day = 0.25）。「この行はニセ」と断定せず、
   **FILE アクセスを APP 再生より弱い視聴シグナルとして扱う**モデル選択（事実として正直）。composite ベースで
   Top50 7〜14 入替・Spearman 0.75〜0.86。可逆・段階調整可。本書の数値を起点に係数を詰める。
2. **C8 availability 設計**（「接続中のみ / 全動画 / 外付け HDD 含む参考ランキング」の分離、`last_known_score` 保存）。
   汚染判定を含まない別軸で、最大の体感ポテンシャル。汚染対策と独立に進められる。
3. **C4（APP_PLAYBACK のみランキング）と現行の比較ビュー**を分析内に追加（本実装ではなく比較用の上限ケース）。
4. **C7（genuine_view_days）の定義を C1 ティアに限定**して再評価。60 秒クラスター版（C7b）は低精度なので採らない。

### 判断の原則（再設計後）

- **「順位が大きく動く」ことを恐れない。** 現状は汚染で歪んでいるので、正常化＝順位が動くこと。
- **恐れるのは「genuine を誤って捨てる（低精度）」と「戻せない（低可逆）」こと。** だから削除系（C3/C4/C5/C7b）は
  本実装しない。
- **効果は減点（C6）と再スコープ（C8）で出す**——精度と可逆を保ったまま順位を動かせる唯一のパターン。

### まだやらないこと

- `FILE_ACCESS_DETECTED` の全除外（C5）／短時間クラスター（C3）の自動除外（低精度・非可逆）
- UI トグルの即追加 / ランキング式の即変更 / DB 履歴の削除・UPDATE

## 18. 採用しない候補と理由

§10.4 の表4 を参照。**どの軸で落ちるか**を明示すると:

- **C2 / C3 は「監査フラグ」として残すが自動除外はしない** — **精度の問題**。C3 は履歴の 67.6% を巻き込み、
  正当な連続視聴を大量に消す。人手で「スキャン副産物」と確証できる部分集合に絞れた場合のみ将来再検討。
- **C4 / C5（FILE 全除外）は比較ビュー専用** — **精度＋可逆性の問題**。過去（2026-01〜06-09）の genuine な視聴も
  二値で全否定する破壊的判断。「本物の FILE 視聴」と「スキャン副産物」を分離する別シグナルが確立するまで本実装しない。
- **C7b は分析比較のみ** — 実質 C3 自動除外で精度が低い。C7 は C1 ティア（=C7a）に限定するなら採用可。
- **式の即変更・UI トグル即追加はしない** — 根拠（C6/C8 の数値）が揃う前に本番挙動を変えると戻せず、
  体感が逆に悪化するリスクがある。

いずれも **「順位が動くから」ではなく「精度が低い／可逆でない／根拠が固まっていない」から**不採用である点に注意。
