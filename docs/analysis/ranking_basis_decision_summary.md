## 0. このレポートの位置づけと安全条件

このレポートは、ClipBox の総合ランキングを巡って実施してきた**複数の分析を 1 本に統合**し、
「**分析から何が分かり、その後どの方針が採用されたか**」を一望できるようにするための文書である。
個々の分析の深掘りは既存の公開資料（§10 参照）に委ね、本書は横断的な結論と意思決定の地図に徹する。

- 分析作成時点では**実装を一切行わなかった**。その後の実装結果は「現在の決定」に明記する。本書の更新では実 DB へ書き込まない。
- 数値はすべて実 DB の**読み取り専用コピー**（`sqlite3` の `mode=ro` URI ＋ `PRAGMA query_only=ON`、`core.database.get_db_connection()` 不使用）から算出した**整数集計のみ**。
- **動画名・ファイル名・フルパス・出演者名・ローカルパス・生 timestamp・匿名対応表は本書に一切含めない。** これらを含む生成物（比較 HTML・時系列 HTML・CSV・コピー DB・匿名対応表）は `docs/analysis/private/`（`.gitignore` 対象）にローカル保持し、公開リポジトリには収録しない。
- 本書だけで話が追えるよう用語は最小限を再掲するが、厳密な定義・手法は `ranking_fairness_multifactor_next_actions.md` に従う。

### 現在の決定（2026-06-21）

分析後、次の方針を採用した。現行仕様の正本は `docs/context/` を優先する。

1. カード、一覧ソート、ランキング、分析などの視聴集計は **`APP_PLAYBACK` のみ**を使用する。
2. 旧 `FILE_ACCESS_DETECTED` / `MANUAL_ENTRY` は集計対象外とし、必要な場合だけ `scripts/purge_legacy_viewing_history.py` で手動削除する。purge は既定dry-runで、全サービス停止、確認文字列、排他ロック、DB/CSVバックアップと検証を必須とする。自動migrationでは削除しない。
3. ランキング画面は **`再生可能だけ / 全動画`** のSelectを持つ。既定は再生可能だけ。全動画では利用不可動画も順位を維持して表示し、論理削除済みは除外する。公開APIは既存の `availability` を維持する。

以下の「現行」「baseline」は、特記がない限り**分析実施時点の表現**である。

## 1. 背景と問い

分析実施時点の ClipBox の総合ランキング（`GET /api/ranking` の composite）には、次の体感劣化があった。

- **「いつも同じ動画が上位に並ぶ」**（顔ぶれが固定）。
- **「期間（今週 / 今月 / 全期間）を変えても順位がほとんど変わらない」**（期間フィルタが効いている実感が薄い）。

ランキングの主変数は `view_days`（= `COUNT(DISTINCT DATE(viewed_at))`、視聴した「日」の数）である。視聴履歴 `viewing_history` の
出所 `viewing_method` には次の 3 種がある。

- **`APP_PLAYBACK`** — 現行アプリでの実再生（単発 `play_video()` と AVP 一括 `record_avp_viewing()`）。**いまも書かれる唯一の方式**で、genuine（本物の視聴）の確実な印。
- **`FILE_ACCESS_DETECTED`** — 旧「ファイルアクセス検知」由来。**現在は書かれない（Phase 1 で archived）**。書き込みは 1 件ずつのループで、各行の `viewed_at` に視聴時刻ではなく**ファイルの `st_atime`（OS 最終アクセス時刻）**を入れていた。
- **`MANUAL_ENTRY`** — 手動入力。本 DB には **0 行**。

当時の `view_days` は method を区別せず数えていた。したがって、もし `FILE_ACCESS_DETECTED` が少数の機械的イベントで大量に水増しされていれば、
(a) 一部の動画だけ不自然に高い `view_days` を持ち続け、(b) その大半が特定の過去日に固まるため期間を絞っても順位が動かない、という
両方の症状を説明できる。

**分析当時の中心の問い**は次の 1 つに集約されていた。

> 総合ランキングの基準を、旧 `FILE_ACCESS_DETECTED` 込みの現行から、**`APP_PLAYBACK`（実再生）ベース**へ寄せるべきか。
> 寄せるとして、**FILE をどう扱い（完全除外か重み下げか）**、**availability（外付け HDD 未接続）の別軸**をどう扱うか。

## 2. 方法とデータ

### 2.1 スコア式（本番一致）

本番 `core/analysis_service.py::get_ranked_videos_for_tab` の composite 式を、コピー DB 上で複製した。

```
composite = int( (view_days · 1 + likes · 3) · (1 + 0.5·t1 + 0.3·t2) · 100 )
```

- `t1` = 判定済み（`current_favorite_level >= 0`）で +50%、`t2` = 選別済み（`is_selection_completed = 1`）で +30%（乗算）。
- `score > 0` のみランキング対象。タイブレーカー `score DESC → last_viewed_at DESC → id ASC`。

### 2.2 「当時のbaseline」と「APP のみ」の定義

| ランキング | view_days / view_count | likes / Tier1 / Tier2 | FILE_ACCESS の扱い |
|---|---|---|---|
| **当時のbaseline** | 全 method 合算 | 当時の実装どおり | スコアに**含む** |
| **APP のみ（現在採用）** | `APP_PLAYBACK` 行のみで再計算 | 係数は同一 | スコアから**除外** |

「APP のみ」は先行分析の候補 **C4/C5（`APP_PLAYBACK_ONLY` / `EXCLUDE_FILE_ACCESS`）** と同義（本 DB は `MANUAL_ENTRY = 0` のため両者は同値）。
**`APP_PLAYBACK` 行（実再生）は 1 件も除外しない。** 除外対象は旧 `FILE_ACCESS_DETECTED` だけである。

### 2.3 スコープ（母集団）

- **再生可能だけ**: `is_available = 1`（いま実際に再生できる動画。現行 UI の既定スコープ）。
- **全動画**: `is_deleted = 0`（外付け HDD 未接続の `is_available = 0` も含む）。

### 2.4 時系列の作り方（本書の新規分析）

`viewing_history` を **日別 × `viewing_method`** で集計し（`viewed_at` は日付に丸め、生 timestamp は出さない）、
`2026-01-01` から最大日まで 0 件日も補完した連続系列を作った。さらに、`FILE_ACCESS_DETECTED` 機能の追加日・archived 化日・
AVP 履歴記録開始日を、リポジトリの commit で確定して縦線イベントとして重ねた（§4.2）。

### 2.5 データ母数

| 項目 | 値 |
|---|---:|
| `viewing_history` 総行数 | 6,725 |
| └ `APP_PLAYBACK` | 1,282（19.1%） |
| └ `FILE_ACCESS_DETECTED` | 5,443（**80.9%**） |
| └ `MANUAL_ENTRY` | 0 |
| └ post-AVP 行（`viewed_at ≥ 2026-06-10`） | 131（**全て `APP_PLAYBACK`**） |
| `videos`（`is_deleted = 0`） | 4,103 |
| └ うち外付け HDD（`storage_location = EXTERNAL_HDD`） | 3,782（**92.2%**） |
| 利用可能（`is_available = 1`） | 221 |
| └ スコア付き（現行 composite > 0） | 208 |
| └ スコア付き（APP のみ composite > 0） | 189 |
| `viewed_at` 範囲 | 2026-01-02 〜 2026-06-18 |

履歴の 5 行に 4 行が旧 `FILE_ACCESS_DETECTED`。AVP 導入後（post-AVP）の 131 行はすべて `APP_PLAYBACK` で、FILE は 1 行も無い
＝ **FILE 汚染は完全に過去側へ閉じている**。

## 3. 仮説（H1–H5）

| ID | 仮説 | 検証指標 |
|---|---|---|
| **H1** | 上位は「実再生」でなく「旧ファイルアクセス日数」で決まっている（FILE 支配） | method 構成比・per-video の app/file 内訳（先行分析 表3） |
| **H2** | FILE 行は少数の日に機械的に集中している（atime バルク） | 日別 FILE 件数の集中度（上位日シェア） |
| **H3** | APP のみへ切り替えると上位が大きく入替する。ただし genuine な APP 実再生は 1 件も捨てない | Top-N 入替・スコア付き本数の変化 |
| **H4** | 体感劣化のもう一つの主因は availability（外付け HDD 未接続による不可視化）で、FILE 問題とは独立 | 再生可↔全動画の入替・Top100 中の利用不可数 |
| **H5** | FILE 完全除外（C4/C5）は精度が低く genuine を巻き込む。重み下げ（C6）が「精度 × 効果」の中庸 | C4/C5 と C6 の Spearman / Top-N 入替の比較（先行分析 表10）。分析上の評価であり、最終採用はAPPのみ |

## 4. 結果

### 4.1 method 構成 — 履歴の 8 割が旧 FILE（H1 の土台）

`FILE_ACCESS_DETECTED` 5,443（80.9%）に対し `APP_PLAYBACK` は 1,282（19.1%）、`MANUAL_ENTRY` は 0。
`view_days` は method を区別せず数えるため、この 8 割の偏りがそのまま順位の主変数に乗る。先行分析の per-video 分解
（`ranking_fairness_multifactor_next_actions.md` 表3）でも、現行 Top50 は全行で `file_view_days ≫ app_view_days` であり、
上位は実再生でなく旧ファイルアクセス日数で並んでいることが確認されている。

### 4.2 時系列 — FILE は少数日に集中し 6 月に消える、APP は全期間に実在（新規）

集計対象期間は **2026-01-01 〜 2026-06-18（169 日、0 件日も補完）**。`FILE_ACCESS_DETECTED` が記録を持つ日は **59 日**、
`APP_PLAYBACK` は **64 日**。

**月別件数:**

| 月 | FILE | APP | 合計 |
|---|---:|---:|---:|
| 2026-01 | 743 | 297 | 1,040 |
| 2026-02 | **1,941** | 249 | 2,190 |
| 2026-03 | 1,007 | 216 | 1,223 |
| 2026-04 | 918 | 129 | 1,047 |
| 2026-05 | 834 | 171 | 1,005 |
| 2026-06 | **0** | 220 | 220 |
| 合計 | 5,443 | 1,282 | 6,725 |

**FILE の日別集中（バルクの指紋）:** 上位日は 2026-02-04 = **703 件**、05-09 = 386、02-05 = 335、02-28 = 275、03-09 = 272。
FILE は **上位 10 日で全体の 54.2%、最大の 1 日だけで 12.9%**（≒700 件）を占める。人が個別に視聴した記録なら 1 日に数百本の別動画が
立つことはあり得ず、これは**少数の機械的イベント（一括アクセスで `st_atime` が揃った）の指紋**である（H2 を支持）。

**APP の日別分布（実再生の形）:** 上位日は 2026-01-03 = 66、03-29 = 66、06-18 = 60、02-28 = 46、02-15 = 45。
APP は **上位 10 日で 37.5%、最大の 1 日でも 5.1%** と、FILE よりはるかになだらかに全期間へ分布する。

**縦線イベント（commit で確定）:**

| 日付 | 種別 | commit | 内容 |
|---|---|---|---|
| 2026-01-02 | 機能追加 | `9efbb20` | 自動ファイルアクセス検知を追加し `FILE_ACCESS_DETECTED` を記録開始 |
| 2026-06-02 | 除外方向 | `aed3378` | Phase 1 整理で FILE/MANUAL を現行機能から除外する方向へ |
| 2026-06-03 | archived | `0b28283` | Phase 1 集約で archived 実体を削除（以降 FILE 検知は現行導線で不動作） |
| 2026-06-10 | AVP 記録開始 | `8bf84fb` | `record_avp_viewing()` で `APP_PLAYBACK` の記録が始まる境界 |

**時系列が新たに見せたこと:** FILE は 2 月（1,941）をピークに 1〜5 月へ偏り、archived 化（06-02/03）の **6 月は 0 件**になる。
一方 APP は 6 月（220 件）も含め**毎月途切れず**存在する。つまり「FILE = 過去に閉じた汚染」「APP = 現在も続く実再生」という構図が、
件数の推移としても裏づけられた。

> 注意: FILE の `viewed_at` は `st_atime` 由来のため、棒の高さは「その日に検知機能が動いた回数」ではなく
> 「その日を atime として持つファイル数」である。バルク日は一括アクセスで atime が揃ったと解釈する。
> APP は実再生イベントそのものなので、件数 = その日の実再生回数。

### 4.3 ランキング入替 — APP のみで Top100 の約 4〜5 割が再編（H3）

利用可能 221 本のうち、APP のみに切り替えるとスコア付きは **208 → 189 本**（FILE だけで支えられていた約 19 本が圏外化）。

**Top-N 入替数**（基準 Top-N のうち候補 Top-N から外れた件数）:

| スコープ | Top50 入替 | Top100 入替 |
|---|---:|---:|
| 再生可能だけ（`is_available=1`） | 21 | 40 |
| 全動画（`is_deleted=0`） | 25 | 47 |

全動画スコープの **25 / 47** は、先行分析（`ranking_fairness_multifactor_next_actions.md` 表10.2 の C4/C5・`all_not_deleted`）の
**25 / 47** と完全一致し、**再現性**を確認した。再生可能だけスコープでも Top100 の **40 件（約 4 割）** が入れ替わる。

**順位移動（再生可能だけスコープ）:**

| 指標 | 件数 |
|---|---:|
| 現行 Top50 にいたが APP のみで Top50 圏外へ | 21 |
| APP のみで新たに Top50 入り | 21 |
| APP のみで +20 位以上 下落（現行 Top100 内） | 46 |

席が「旧ファイルアクセスで押し上がっていた動画」から「実際にアプリで再生されている動画」へ動く。なお **`APP_PLAYBACK` 行は 1 件も除外していない**
ので、順位が動くのは「実再生で稼いだ分が消える」からではなく「旧ファイルアクセスの水増しが外れる」からである。

### 4.4 availability — FILE とは独立した第二の体感要因（H4）

| 指標 | 件数 |
|---|---:|
| ライブラリ 4,103 本のうち外付け HDD | 3,782（92.2%） |
| 本番で可視（`is_available = 1`） | 221（スコア付き 208） |
| 「再生可能だけ」Top50 ↔「全動画」Top50（APP のみ）の入替 | 30 |
| 「全動画」APP のみ Top100 のうち `is_available = 0`（利用不可） | 30 |

ライブラリの **92.2%** が外付け HDD にあり、未接続だと `is_available = 0` でランキングから消える。全動画 APP Top100 の 30 件が利用不可
＝ **可視ランキングは実質「いま接続中のストレージ内ランキング」**になっている。これは FILE 汚染とはまったく別の軸であり、
FILE を直しても接続状態でランキングの見え方が激変する余地が残る。

### 4.5 C1 高信頼バルク

C1 = `EXACT_TIMESTAMP_GE5_LEGACY_FILE_ACCESS`（同一 `viewed_at` 完全一致 `group_size ≥ 5` ∧ `FILE_ACCESS_DETECTED` ∧
`viewed_at < 2026-06-10`）は、本 DB で **11 群 / 63 行 / 61 動画・`APP_PLAYBACK` 混入 0**。5 件以上が秒単位で一致するのは
個別視聴では作れず、1 回の機械的ファイル操作の指紋である。**ただし C1 だけを隔離しても上位はほぼ動かない**（先行分析: Top50 入替 0・
Spearman 0.996）。これは「C1 が安全だから」ではなく「明白バルクが上位汚染のごく一部しか占めない」ためで、C1 は体感改善策ではなく
**衛生（hygiene）＋回帰テストの土台 / 将来の `bulk_reason` 監査フラグ**として位置づけるのが正しい。

## 5. 考察

### 5.1 仮説の判定

- **H1（FILE 支配）: 支持。** 履歴の 80.9% が FILE で、per-video でも上位は `file_view_days ≫ app_view_days`。
  「いつも同じ・期間で動かない」体感の中心原因。**＝現状ランキングは“正しい状態”ではなく、修正で順位が動くことは正常化である。**
- **H2（FILE はバルク集中）: 支持。** FILE は 59 日に集中し上位 10 日で 54.2%・1 日最大 12.9%。APP（上位 10 日 37.5%・1 日最大 5.1%）と
  対照的で、時系列でも 1〜5 月に偏り 6 月に 0。少数の機械的イベントが多数の「視聴日」に水増しされている。
- **H3（APP のみで大入替・genuine は無傷）: 支持。** Top100 が約 4〜5 割入替し、スコア付きは 208→189。除外対象は旧 FILE のみで
  APP 実再生は 1 件も捨てていない。
- **H4（availability も主因・独立）: 支持。** 92.2% が外付け HDD、全動画 APP Top100 の 30 件が利用不可。FILE 汚染とは独立の軸。
- **H5（完全除外は精度低・C6 が中庸）: 分析上は支持。ただし最終採用はAPPのみ。** 先行分析の評価軸（精度・可逆性・効果）では、C4/C5（FILE 全除外）は
  Spearman が composite で 0.6〜0.82・生 view_days では最低 0.23 まで落ちる一方、**genuine な過去の FILE 視聴ごと消す（精度が最低・段階調整不能）**。
  C6（FILE 日の重み下げ・除外しない）は composite で Spearman 0.75〜0.86・Top50 入替 12〜14 を保ちつつ、係数で**可逆**に調整でき
  「この行はニセ」と断定しない。当時の分析推奨は **C1（衛生）→ C6（体感改善の本命）＋ C8（availability）**だったが、最終的には単純で監査可能なAPPのみ集計を採用した。

### 5.2 二つの方針が公開資料の中で交差していた

- `ranking_app_playback_comparison_findings.md` は、ユーザー方針として **APP のみ（= C4/C5・FILE をスコアから無視）**を確定路線として記録し、
  「genuine な実再生を保ったまま Top100 の約 4〜5 割を再編＝意味のある正常化」と結論している。
- 一方 `ranking_fairness_multifactor_next_actions.md` は、精度の観点から **C4/C5 を本採用には推さず**、**C6（重み下げ）**を体感改善の本命に置く。

両者は、**「実再生だけを信じる単純さ・可読性（A: 完全除外）」と「genuine な FILE 視聴を弱シグナルとして残す精度・可逆性（B: 重み下げ）」の
トレードオフ**を別角度から述べていた。最終決定では A を採用し、現行の視聴集計をAPPのみに統一した。係数による重み下げや行単位の除外フラグは導入していない。

### 5.3 時系列が意思決定に足した一押し

時系列は「FILE = 過去に閉じたバルク（6 月に 0）」「APP = 現在も毎月続く実再生」を件数で可視化した。これは
**「APP ベースへ寄せる」という大方向の妥当性を補強**する。同時に、APP のみでもスコア付きが 208→189 にしか減らない
（土台は十分ある）ことも示しており、**移行しても上位が空洞化しない**ことの裏づけになっている。

## 6. 判断してよいこと（データの結論）

ここまでの分析から、次の 4 点は**データに基づいて結論として扱ってよい**。

1. **現状維持は選択肢から外してよい。** 当時のランキングは旧 FILE（履歴の 80.9%）に支配されており、上位固定化の主因。
   「壊れた順位を保存する」ことに体感上の価値はない。
2. **APP ベースへ寄せる大方向は妥当。** APP のみで Top100 の約 4〜5 割が入替＝効く。しかも APP は時系列上、現在も毎月続く実再生で、
   移行しても上位は空洞化しない（208→189）。
3. **この移行は確実な実再生（APP）を犠牲にしない。** 旧 FILE に genuine が混じる可能性は分析上残るが、現行の信頼できる集計基準には使わない。旧履歴は自動削除せず、明示的な手動purgeを選んだ場合だけ検証付きで削除する。
4. **availability は独立した第二の大問題。** ライブラリの 92.2% が外付け HDD で、可視ランキングは実質「接続中ストレージ内ランキング」。
   FILE を直すだけでは解決しない別軸が残っている。

---

## 7. 現在の決定

### 7.1 FILE の扱い — Aを採用済み

| 案 | 内容 | 向き / 不向き |
|---|---|---|
| **A: 完全除外**（APP のみ・C4/C5） | FILE は集計対象外。実再生だけ信頼 | **採用済み**。単純で全集計に一貫して適用できる |
| **B: 重み下げ**（C6） | FILE 日を減点（除外しない）。弱いシグナルとして残す | 不採用。係数と集計定義を増やさない |

分析値は当時の判断材料として保持する。現行実装は `viewing_method = 'APP_PLAYBACK'` を明示的に条件へ加え、旧methodを全視聴集計から除外している。

### 7.2 旧履歴の扱い — 安全な手動purgeを採用済み

- 分析当時の「履歴は削除しない」は、比較を安全・可逆に行うための前提だった。
- 現在は `scripts/purge_legacy_viewing_history.py` を導入し、旧2methodを削除できる。ただし**自動実行しない**。
- 実行時はサービス停止、確認文字列、排他ロック、DB/CSVバックアップ、整合性・件数・SHA-256検証を必須とし、`APP_PLAYBACK`、`NULL`、未知methodは保持する。

### 7.3 availability — 表示範囲Selectを採用済み

- UIは `再生可能だけ / 全動画` を明示的に切り替えるSelectを持ち、既定は再生可能だけ。
- 全動画では利用不可動画もスコア順のまま含める。論理削除済みは常に除外する。
- 公開APIは既存の `availability=利用可能のみ|利用不可のみ|すべて` を維持し、破壊的な`scope`置換は行わない。

### 7.4 採用しなかった案

- FILEの重み付け、`bulk_reason`等の除外フラグ列、ランキング係数の変更は行わない。
- 「発掘 / 新鮮味 / 最近見てない」は総合ランキングに混ぜない（必要なら別機能・別導線）。
- 短時間クラスター（C3）等の自動除外はしない（genuine を大量に巻き込む＝精度が低い）。

1. APPのみ集計への統一は完了。

3. ランキングの表示範囲Selectは完了。
