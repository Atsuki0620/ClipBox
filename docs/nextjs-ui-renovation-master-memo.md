## 3-A. Phase 0 決定事項（2026-06-23）

Phase 0 は実装前の方針整理であり、本体画面・API・DB・設定ファイル・実データは変更しない。ここでの「決定」は UI 改修の進め方に対する決定で、`SPEC_NEXTJS.md` などの正本仕様を上書きしない。

### 画面単位の扱い
- 現行の8画面（Tier1 / Tier2 / あとで見る / AVP / ランキング / 分析 / 検索 / 設定）を維持する。用途別に大きく再編した新ナビゲーションは作らない。
- Tier1/Tier2 はどちらも「ライブラリ / ランダム / 運命の1本」の3タブを維持する。完全共通 UI に寄せ切るのではなく、カード・レベルボタン・再生・いいね・あとで見る・AVP候補・グリッド・空/読込/エラー状態は共有候補、タイトル・状態文言・フィルタ・空状態説明・主操作の意味は Tier 別に最適化する。
- ラボ案は「見た目の検討材料」であり、実装時は本体の状態境界（DB / localStorage / sessionStorage / メモリ）と `displayContext` 3値固定を優先する。

### 画面別の比較方針
- あとで見るは独立メニュー・3セクションを維持する。2026-06-27時点のフィードバックでは案B「付与理由・状態説明強化型」ベース、案Cの上部サマリー不採用、タイトル横 Tooltip に解除ルールを集約、処理済み候補は個別解除基本とする。詳細は `docs/nextjs-ui-renovation-feedback.md` に集約する。
- AVP は候補一覧中心 / 左右分割 / 上下分割 / 候補管理・再生対象タブ / 上下分割リッチカードの各案を比較した。2026-06-27時点のフィードバックでは案A〜D をそのまま採用せず、案Dを発展させた案D改「上段候補テーブル / 下段再生セット2×2カード」を採用候補とする。上段はランキング/検索と整合した候補テーブル、下段は Tier1 ライブラリ動画カードと整合した2×2再生セットカードにする。候補上限なし・再生対象最大4本・AVP候補/再生対象の状態境界・`displayContext` 3値固定は不変。詳細は `docs/nextjs-ui-renovation-feedback.md` の AVP 欄に集約する。
- ランキングはカードランキング / テーブル / カード＋数値順位 / 上位カード＋下位テーブルを比較対象にした。2026-06-26 時点のフィードバックでは A/B/C を不採用とし、案D「操作付きスコアテーブル」を新方針とする。総合スコア式・APP_PLAYBACK 基準・タイブレークは変更しない。
- 分析は UI LAB A/B/C の採用判断対象から外し、実データ体験ベースの別枠フィードバックへ移す。Recharts と既存集計前提は維持し、詳細は `docs/nextjs-ui-renovation-feedback.md` に集約する。
- 検索は現状改善型 / Tier1・Tier2カード整合型 / 高機能フィルタ型 / 検索結果作業台型を比較対象にした。2026-06-26 時点のフィードバックでは案D「高機能フィルタ + 操作付きテーブル」を採用候補とし、ランキングとは統合せず共通の操作付きテーブル土台だけ共有する。検索結果を別状態として永続化しない。
- Runtime control はサイドバー下部維持を前提に、配置を変えず中身を整理する（2026-06-27）。Streamlit の状態表示・停止ボタンを UI から削除し、FastAPI / Next.js を個別表示（状態ランプ/起動状態/ポート/最終確認時刻）。停止操作は `アプリを停止` の1ボタンにまとめる。設定画面へ移設しない・既定無効・Streamlit 関連コードの削除可否は実装時調査。
- Tier2 は専用 UI を決め切らず、案1「Tier1 Variant K 流用案」と案2「Tier2専用文言強め案」を比較する（2026-06-27）。カード/操作部品は原則 Tier1 流用、専用化候補は見出し・空状態・フィルタ文言など文章部分のみ。該当Tierバッジは `Tier2` のみ（Lv・未選別/選別済みは書かない）、初期フィルタは未選別+選別済み、フィルタ候補 すべて/未選別/選別済み・既定すべて。重い専用 UI LAB の前に軽い比較で判断する。
- 設定は scan-first 系のうち上部タブ案を採用候補とする（2026-06-27）。スキャンは バックアップ → Tier1 → Tier2 の3段を維持し、進捗バー・経過時間・完了後の所要時間・結果詳細の折りたたみを追加したい。カード表示設定は **メタ項目**（タイトル下の補足情報）と **バッジ項目**（状態ラベル）に分け、該当Tierバッジを新設する。内部 config キー・バックアップガード・スキャン API は変更しない。詳細は `docs/nextjs-ui-renovation-feedback.md` の設定欄に集約する。
- 全画面共通方針（2026-06-27）: 視聴回数ではなく **視聴日数** に一本化し、更新日・登録日は全 UI から外す。使う日付は 作成日（動画ファイルの作成日 `file_created_at`）/ 判定日 / 選別日 を基本にする。視聴履歴・DB・API・ランキング計算式は変更しない。詳細は `docs/nextjs-ui-renovation-feedback.md` の全画面共通フィードバックに集約する。

### 採用方針として次回実装で反映するもの
- **AVP再生であとで見るを自動解除しない**方針を採用する。現行実装では処理済み動画の AVP 起動成功で `watch_later` が解除されるため、次回実装で正本・API・テストを同時に合わせる。
- 自動解除は以下に限定する: 判定/選別完了、処理済み動画へのいいね。
- 自動解除しない操作は以下のまま維持する: 通常再生、AVP候補追加、AVP再生対象チェック。
- 次回実装で変更する操作: AVP起動成功・AVP再生履歴記録では `watch_later` を解除しない。
- 設定の上部タブ版主案化は UI 方針であり、本時点では `user_config.json` のキー、バックアップガード、スキャン API、履歴データ設計を変更しない。

### 初心者向け補足
- **UI LAB**: `frontend/src/app/lab/` にあるモック画面。実 DB/API/localStorage へ接続しない比較用で、本体導線ではない。
- **KPI**: 未判定数・判定率など、画面上部で状態を短く示す数値カード。
- **localStorage**: ブラウザに保存する状態。AVP候補や再生対象のように「そのブラウザだけでよい」状態に使う。
- **sessionStorage**: ブラウザタブのセッション中だけ残る状態。運命の1本の保持に使う。
- **採用方針（未実装）**: 次回実装で反映することを決めたが、まだ本体コードには入っていない方針。正本ドキュメント・テスト・受け入れ基準を同時に変える必要がある。

## 3-C. 今後の作業フロー（UI LAB → 全画面フィードバック → 統合 Variant K → サンプルDB → 実DB統合）

Phase 0 と同様、ここでの「作業フロー」は UI改修の進め方の整理であり、本体画面・API・DB・設定ファイル・実データは変更しない。各画面のフィードバックは `docs/nextjs-ui-renovation-feedback.md` に蓄積する。

### 作業フロー

1. **UI LAB不足画面を追加検討する**: ランキング / 検索。既存の「あとで見る」「AVP」の UI LAB 結果もフィードバック対象に含める。分析は別枠で扱う。
2. **全画面の UI LAB フィードバックを一巡させる**: Tier1 / Tier2 / あとで見る / AVP / ランキング / 検索 / 設定 / Runtime control。分析は UI LAB 採用判断の完了条件から外し、実データ体験ベースの別枠フィードバックにする。
3. **全画面統合版 Variant K を作成する**: 各画面の単発案ではなく、ClipBox 全体で整合した次期UI案として作る。ナビゲーション・カード・余白・ボタン・状態バッジ・情報密度を揃える。
4. **実DBを匿名化してサンプルDBを作成する**: 実動画名・実パス・実フォルダ名・個人情報につながる文字列は残さない。ただし、実際の件数感・レベル分布・視聴履歴・あとで見る・ランキング・利用不可状態など、UI確認に必要な状態は残す。
5. **サンプルDB接続の Variant K 版 ClipBox を作成する**: 実DBを壊さず、実アプリに近い操作感を確認するため。サンプル確認用 Variant K 版 ClipBox での試用フィードバックも `docs/nextjs-ui-renovation-feedback.md` に記録する。
6. **フィードバックを受けてサンプル確認用 Variant K 版 ClipBox を改善する**。
7. **問題なければ実DB接続版へ進む**: 実DB接続時は本体 ClipBox を停止しておく。本体 ClipBox と Variant K 版 ClipBox が同じ実DBへ同時書き込みできる状態にはしない。
8. **安定確認後、本体 ClipBox へ統合する**: 不具合時にすぐ本体 ClipBox へ戻せるよう、安定版を残す。実装・移行は段階的に行う。

### 並列運用の注意点

- サンプル確認用 Variant K 版 ClipBox は、本体 ClipBox と並列で確認できるようにしたい。
- ただし、同じ実DBへの同時書き込みは避ける。
- Variant K を実DBに接続して確認する場合は、本体 ClipBox を停止する。
- 目的は、Variant K に不具合があった場合に、安定版の本体 ClipBox へすぐ戻れるようにすること。
- DB破損・履歴汚染・settings破損・ランキング結果への悪影響を防ぐことを優先する。

> 注: この並列運用方針は `CLAUDE.md` 設計原則5（書き込みは一方のサーバーのみ。WAL 未設定のため同時書き込みは `database is locked` / `SQLITE_BUSY` 相当で失敗し得る）と整合する。サンプルDB匿名化方針（実動画名・パス・フォルダ名・個人情報を残さない／件数感・レベル分布・視聴履歴・あとで見る・ランキング・利用不可状態は残す）は上記ステップ4を参照。

## 6. 今後の推奨ロードマップ

「正本メモ作成 → 画面別仕様整理 → 小さな UI 実装 → 受け入れ確認」の順。いきなり大規模実装しない。本流と混ざらないよう、実験は専用ブランチ／git worktree で進める。小さく戻せる単位にする。

### 短期

#### S1. 文書整合（本タスク）
- **目的**: 現在地と確定/未決を1枚に集約し、以降の判断基準にする。
- **なぜこの順番**: 提案を確定と誤認したまま実装に入るのが最大のリスク。先に棚卸し。
- **対象**: `docs/nextjs-ui-renovation-master-memo.md`（本ファイル）。
- **やらないこと**: コード・データ・設定変更、スキャン、バックアップ。
- **確認方法**: `git diff`、Pull request の略語表記が混入していないかのチェック、フルパス/個人情報チェック。
- **Pull request 単位**: 文書のみで1本。

#### S2. 画面別「現行↔ラボ」差分の棚卸し
- **目的**: ラボのどの提案を本体に持ち込むかを画面ごとに表化し、要再確認項目を1つずつ意思決定可能にする。
- **なぜこの順番**: 実装前に「採用/保留/不採用」を確定させ、Prompt 単位を切り出せるようにする。
- **対象**: `docs/`（本メモ §3・§4 を肉付け、または別表）。
- **やらないこと**: 実装。
- **確認方法**: ラボの各 `COMPARISON_*.md` と本体実装の突き合わせ（読み取りのみ）。
- **Pull request 単位**: 文書のみで1本。

#### S3. UI検討用 worktree 整備
- **目的**: 本流を汚さずに本体への小実験を回す土台。
- **なぜこの順番**: 実装に入る前に隔離環境を用意し、いつでも破棄できるようにする。
- **対象**: git worktree（専用ブランチ）。
- **やらないこと**: main への直接実装。
- **確認方法**: `git worktree list`、ビルド起動確認。
- **Pull request 単位**: 環境準備のみ（コミット不要なら不要）。

#### S4. 主要画面1つ（Tier1 ライブラリ）でのUI方向性 小実験
- **目的**: Variant J 方向（ツールバー再構成・Popover 化）を本体で最小実装し、方向性を体感で確定。
- **なぜこの順番**: ライブラリは最頻出かつ Tier2 へ横展開できる雛形。1画面で型を作る。
- **対象**: `LibraryWorkspace.tsx` / `FilterPanel.tsx` / `LibraryFilterBar.tsx`。
- **やらないこと**: フィルタ/ソートの永続化、`displayContext` 拡張、データ取得ロジックの変更。
- **確認方法**: `npm run lint` + `npm run typecheck` ＋ 手動スモーク（必要なら Playwright で画面確認）。
- **Pull request 単位**: 「見た目（ツールバー再構成）」で1本。仕様変更（あとで見るフィルタ廃止など）は別 Pull request に分離。

### 中期

#### M1. 共通レイアウト統一（サイドバー見出しグルーピング等）
- **目的**: 全画面に効く土台の見た目を1回で整える。
- **なぜこの順番**: Tier1 実験で型が決まってから全画面に波及させると手戻りが少ない。
- **対象**: `SidebarNav.tsx` / `layout.tsx` / 共通カード。
- **やらないこと**: ナビ項目数・遷移先・ルート構成の変更。

- **Pull request 単位**: サイドバーで1本、共通カードで1本。

- **なぜこの順番**: 自動保存・スキャン1操作化・履歴記録は新規実装でリスクが高いため、まず命名/レイアウトだけを切り出す。単一カラム/カテゴリレールは比較参考に留める。
- **対象**: `frontend/src/app/settings/page.tsx`（＋自動保存/履歴の新設計）。
- **やらないこと**: 内部 config キーの変更、バックアップガード（409）の緩和、スキャン/バックアップの実行を伴う検証は慎重に。
- **確認方法**: lint/typecheck ＋ 設定保存・スキャンガードの手動確認（破壊操作は避ける）。
- **Pull request 単位**: ①命名/レイアウト（見た目）→②自動保存→③スキャン1操作化＋履歴、の段階分割。

#### M3. あとで見る画面の磨き込み
- **目的**: 既に独立済みの画面を、案Bベースの付与理由・状態説明強化型へ整合。
- **対象**: `watch-later/page.tsx` / `VideoCard.tsx`。
- **やらないこと**: #45 の反映安定化最適化を崩さない、DB 永続境界を変えない。
- **確認方法**: あとで見るトグルの反映確認（手動）。
- **Pull request 単位**: 見た目整合で1本。

#### M4. ranking / AVP の順次見直し（analysis は別枠）
- **目的**: 残り画面を共通方針へ波及。
- **なぜこの順番**: ranking は公平化の計算成果に触れない見た目のみ、AVP は最後。analysis は UI LAB 採用判断ではなく実データ体験ベースの別枠で扱う。
- **対象**: `/ranking` と `/avp` の各 `page.tsx`。analysis は別枠で扱う。
- **やらないこと**: ランキング計算/指標源（APP_PLAYBACK）の変更、AVP の localStorage 境界・4件上限の変更。
- **確認方法**: lint/typecheck ＋ 各画面スモーク。
- **Pull request 単位**: 画面ごとに1本。

各段の共通確認: `npm run lint` + `npm run typecheck`（自動テストは無い）＋ `docs/context/AI_WORKFLOW.md` の手動スモーク。必要に応じ Playwright で画面確認。

## 7. 実装前にClaude Code / Codex へ渡すPrompt方針

- **1回の Prompt で広げすぎない**: 1画面・1テーマに絞る。複数画面を一度に変えない。
- **画面単位で分ける**: 「Tier1 ライブラリのツールバー」など対象を限定する。
- **仕様変更と見た目変更を混ぜない**: 見た目（再配置・色・余白）と挙動変更（フィルタ廃止・保存方式変更）は別 Prompt・別 Pull request にする。
- **実データ変更を避ける**: DB 書き込み・スキャン・バックアップ・設定ファイル書き換えを伴う検証は明示的に禁止・代替（モック/手動）を指示する。
- **壊しやすい点を必ず明記**: 状態の永続境界（DB/localStorage/sessionStorage/メモリ）、`displayContext` 3値固定、API は snake_case、フィルタ/ソートは非永続、ライブラリスキャンのバックアップガード、`scan_and_update` の全件 `is_available=0` 化。
- **受け入れ確認項目を必ず含める**: `npm run lint` + `npm run typecheck` ＋ 当該画面の手動スモーク手順、必要なら Playwright の確認観点。
- **Pull request 要約に含める内容**: 目的 / 変更点（見た目か仕様か）/ 触っていない境界の明記 / 確認したコマンドと結果 / スクリーンショットまたはスモーク結果 / 戻し方。
