## 1. 統合 Variant K の目的

画面ごとに個別検討してきた UI 案を、ClipBox 全体で **ナビ・カード・余白・ボタン・状態バッジ・情報密度・テーブル/カードの使い分け・用語** を 1 つに揃えた統合方針として整理する。

- Variant J 系の **寒色モダン** な方向性をベースにする。
- Tier1 で固めた **variant-k** の考え方を全画面へ広げる。
- ただし、まだ本体実装ではない。**サンプルDB接続前の UI LAB モック** に向けた準備である。
- 確認したいのは、全画面の **整合感・情報密度・操作感・用語・バッジ・テーブル/カードの使い分け**。
- Tier2 はまだ確定しきらず、**Tier1流用案と専用文言案を見比べたい**。
- あとで見る・AVP・ランキング・検索・設定・Runtime control は、すでに画面別に決めた案を統合 Variant K に反映したい。

本書は実装計画書とワイヤー方針メモのみを成果物とする。

## 5. UI LAB ディレクトリ構成案

統合 Variant K を置く LAB のディレクトリ構成を 3 案比較する。観点は、統合シェルの置き場／既存 tier1 variant-k との関係／既存 Variant A〜K を壊さないか／画面間移動で統合感を確認しやすいか／サンプルDB接続版へ進みやすいか／実装段階を分けやすいか／スクショ・レビュー資料を `_review/` に置きやすいか。

### 案A: 単一ルート + 画面内スイッチャ
`frontend/src/app/lab/variant-k/page.tsx` 1 枚にクライアント state の画面スイッチャを持たせ、配下に `_components/`・`_data/`・`_review/` を置く。

- **メリット**: 1 URL で全画面を横断でき統合感を即確認／状態共有が容易／ファイル数が少ない。
- **デメリット**: URL が画面ごとに分かれず、Playwright で各画面の全体スクショを安定ターゲットにしづらい／1 ファイルが肥大化しやすい／サンプルDB版でルートを切り直す手戻りが出やすい。
- **推奨度**: 中。

### 案B（推奨）: マルチルート + 統合シェル `layout.tsx`
`frontend/src/app/lab/variant-k/<screen>/page.tsx`（screen = `tier1` / `tier2` / `watch-later` / `avp` / `ranking` / `search` / `settings` / `analysis`）に分け、`frontend/src/app/lab/variant-k/layout.tsx` に統合シェル（モックサイドバー常駐＋Runtime control サイドバー下部）を置く。共通部品は `frontend/src/app/lab/variant-k/_components/`、モックは `_data/`、スクショ資料は `frontend/src/app/lab/variant-k/_review/`。

- **メリット**: Next.js ルーティング idiom に沿う／画面ごとに安定 URL を持ち `_review/` へスクショを取りやすい／統合シェル（ナビ・サイドバー・カード共通化）を素直に表現できる／サンプルDB版へ拡張しやすい／既存 A〜K を壊さない／本書の置き場 `lab/variant-k/_review/` と整合。
- **デメリット**: ファイル数が増える／`layout.tsx` の責務設計が必要。
- **推奨度**: 高。

### 案C: 最上位ルートグループ `app/(variant-k)/`
`frontend/src/app/(variant-k)/` のルートグループにして、本体の `SidebarNav` を迂回し完全独立シェルを作る。

- **メリット**: 本体サイドバーの外で独立シェルを構築できる。
- **デメリット**: lab 規約（`LabFrame`・`Modern*` 等の共有資産）から外れる／レビュー導線が `/lab` ハブから切れる／本体に近すぎて「現行」と誤認するリスク。
- **推奨度**: 低。

## 6. 推奨ディレクトリ構成と、その理由

**案B を推奨する。**

理由:

- 既存の「完成 Variant」idiom（`tier1-library/variant-j/` が `page.tsx` ＋ 分割サブ部品 ＋ `shared.ts`）の自然な延長になる。画面横断は `layout.tsx` 統合シェル ＋ `lab/variant-k/_components/`（lab 全体共有の `LabFrame` / `Modern*` を再利用、または後方互換 optional props でフォーク）にまとめられる。
- 各画面が安定 URL を持つため、Playwright での全体スクショと `_review/` への保存がやりやすい。
- 統合シェル（サイドバー常駐・画面間移動）を素直に表現でき、統合感の確認という目的に最も合う。
- サンプルDB接続版へ進むとき、画面単位でデータ接続へ差し替えやすい。
- 本書の置き場 `frontend/src/app/lab/variant-k/_review/INTEGRATED_VARIANT_K_PLAN.md` がすでに `lab/variant-k/` を前提にしており一貫する。

> **今回はディレクトリ構成を決めるだけで、実装はしない。** 最終確定は §16 のユーザー確認に回す。

### 旧 Tier1 variant-k 指示との関係（誤読防止）

feedback.md 付録の **Tier1 variant-k 実装指示** は、`/lab/tier1-library/variant-k`・`/lab/tier1-random/variant-k`・`/lab/tier1-fate/variant-k` という **画面別エリア配下の variant-k**（＝Tier1 向け単独 LAB 案）を想定している。

一方、本書の **統合 Variant K（案B）** は `/lab/variant-k/<screen>` という **全画面横断の統合シェル配下** に Tier1 を含む各画面を置く別構成である。両者は名前が似るが、配置・目的が異なる。

統合 Variant K の Tier1 画面は、旧 Tier1 variant-k 付録の設計内容（2行タイトル・作成日表示・ランダム固定条件・運命の1本の履歴撤去 等）を **設計指針として流用** するが、旧 `/lab/tier1-*/variant-k` ルートを移設・改変するものではない。旧ルートを別途作るか作らないかは段階3でユーザー確認する。

用語上は、`variant-k` = 既存 Tier1 向け LAB 案、`統合 Variant K` = 全画面整合 UI 案、として区別する。

## 8. 全画面共通ルール

統合 Variant K の基本方針。

- **視聴回数ではなく視聴日数を主役にする。** カード/テーブルから視聴回数を外す。
- **更新日・登録日は出さない。** 日付は **作成日 / 判定日 / 選別日** を基本にする。
  - 作成日 = 動画ファイルの作成日（`file_created_at`）。ClipBox 登録日時 `created_at` とは別。
  - Tier1 カードでは **判定日**、Tier2 カードでは **選別日**。
- **該当Tierバッジ** は **Tier1 / Tier2 の Tier 名のみ** を表示する。Lv や 未判定 / 未選別 は書かない。**初期 ON。**
- **初期 ON のバッジ** は **該当Tier / あとで見る / 利用不可** を基本にする。
- **AVP候補バッジは初期 OFF**（現行フィードバックで「初期 ON にしない」と記録済み）。初期 ON へ覆す場合は §16 の「既定を覆す確認項目」に回す。
- **再生中はバッジよりハイライトを優先** する。再生中ハイライトは **amber 系を暫定値** とする。
- **利用不可** は、テーブルでは行を薄く表示し、カードではバッジ＋薄表示にする。利用不可の動画では **再生操作や再生対象追加操作ができないことを見た目で分かる** ようにする。
- **テーブル優先画面** = ランキング / 検索 / AVP候補。**カード優先画面** = Tier1 / Tier2 / あとで見る / AVP再生セット。
- 長い説明は **タイトル横の Tooltip** へ集約する。主要ボタンに Tooltip を増やしすぎない。
- `displayContext` は **`tier1` / `tier2` / `avp` の3値固定**。
- **あとで見ると AVP候補を混同しない。**
- 同じ概念は画面間で同じ見た目にする（視聴日数 / 該当Tier / あとで見る / 利用不可 をカードとテーブルで一致させる）。

## 9. 壊してはいけない既存仕様

モック計画でも前提にする（`SPEC_NEXTJS.md` §0/§14、`AI_WORKFLOW.md` §D）。

- **永続境界を越えない。**
  - あとで見る / レベル / いいね = **DB 相当**。
  - AVP候補 / 再生セット / 運命の1本の保持 = **localStorage / sessionStorage 相当**（AVP候補・再生対象・再生中ハイライト=localStorage、運命の1本の保持=sessionStorage）。
  - フィルタ / ソート = **メモリ相当**（永続しない）。
- Tier1 は **セレクション操作を常に出さない。**
- Tier2 のレベル操作は既存仕様に沿う（ドロップダウンは「未選別 / Lv0..Lv4」、未判定 -1 は出さない）。
- **総合スコア式・タイブレーク・APP_PLAYBACK 基準** は変更しない。
- AVP の **候補上限なし**、**再生対象は最大4本**、**欠落ID掃除**、**再生後クリア**。
- **運命の1本の保持は残す。** 履歴セクション撤去は、保持仕様の撤去ではない。

## 10. 画面別ワイヤー方針

実装はせず、文章で方針を整理する。

### Tier1
- **目的**: 未判定動画に初めてレベルを付ける一次判定。
- **主な表示領域**: KPI パネル ＋ タブ（ライブラリ / ランダム / 運命の1本）＋ カードグリッド/テーブル。
- **カード優先/テーブル優先**: カード優先（ライブラリはカード⇄テーブル切替を内包）。
- **主役にする情報**: タイトル（2行）・視聴日数・作成日・判定日。
- **出さない情報**: 視聴回数・更新日・登録日。
- **操作ボタン**: 再生・レベル付与・いいね・あとで見る・AVP候補。**セレクション操作は出さない。**
- **バッジ**: 該当Tier（Tier1）・あとで見る・利用不可を初期 ON。判定済み/利用不可は薄表示。
- **Tooltip に逃がす説明**: タブの意味・ランダム/運命の条件説明。
- **空状態**: 「未判定の動画がありません」等、判定対象が尽きた旨。
- **利用不可状態**: カードはバッジ＋薄表示、再生/AVP候補追加を不可表示。
- **あとで見るとの関係**: あとで見る付与/解除ボタンを持つ（DB 相当）。
- **AVP候補との関係**: AVP候補追加ボタンを持つ（localStorage 相当、あとで見るとは別概念）。
- **他画面と共通化すべき部品**: カード（Tier2・あとで見る・AVP再生セットと共通）、レベルボタン、操作ボタン群。
- **個別方針（variant-k 付録土台）**: 見出しは「Tier1」のみ・サブタイトル削除／2行タイトル（2行分の min-height 確保）／作成日を「作成 yyyy/mm/dd」表示／**ランダムは「未判定かつ再生可能」固定**（条件パネル撤去）／**運命の1本は履歴セクション（前回引いた1本）を撤去**し、大型の「引く」ボタンを中心に据える／`displayContext="tier1"` を守る。

### Tier2
- **目的**: 判定済み動画から選別フォルダへ選び、選別状態（未選別 `!` / 選別済み `+`）を扱う二次判定。
- **主な表示領域**: Tier1 と同型のタブ ＋ カードグリッド ＋ **案1/案2 トグル**。
- **カード優先/テーブル優先**: カード優先。
- **主役にする情報**: タイトル・視聴日数・**選別日**・レベル。
- **出さない情報**: 視聴回数・更新日・登録日。
- **操作ボタン**: 再生・レベル操作（未選別/Lv0..Lv4）・いいね・あとで見る・AVP候補。
- **バッジ**: **該当Tier は Tier2 のみ**（Lv・未選別/選別済みは書かない）。
- **Tooltip に逃がす説明**: 未選別/選別済みの意味・選別後の次操作。
- **空状態**: Tier2 専用文言の候補（案2 で強化）。
- **利用不可状態**: Tier1 と同じ扱い。
- **あとで見る/AVP候補との関係**: Tier1 と同じ。
- **他画面と共通化すべき部品**: カード/操作部品は Tier1 から流用。文言のみ Tier2 専用化候補。
- **初期フィルタ**: 未選別 ＋ 選別済み。候補 = すべて / 未選別 / 選別済み（既定すべて）。
- **2案トグル（同一画面内）**:
  - **案1 Tier1流用案**: Tier1 Variant K のカード/操作部品/余白/グリッド/レイアウトをほぼ流用し、**文言のみ変更**。
  - **案2 Tier2専用文言強め案**: カード/操作部品は Tier1 流用のまま、**見出し・タブ説明・空状態・フィルタ文言・未選別/選別済み説明・選別後の次操作文言** を Tier2 専用に強化。
  - **両案の差分** は「文言レイヤーのみが異なり、カード/操作/グリッドは共通」である点を明示する。
  - **ユーザー確認に回す判断**: 案1で十分か、案2の専用文言が必要か。

### あとで見る
- **目的**: 判定/選別を後回しにした動画を消化する独立タスク画面。
- **主な表示領域**: **未処理 / 確認・見直し / 処理済み候補** の3セクション。
- **カード優先/テーブル優先**: カード優先（共通カードを再利用、サムネ無し・情報カードのみ）。
- **主役にする情報**: タイトル・視聴日数・状態（ステータス）・付与理由。
- **出さない情報**: 視聴回数・更新日・登録日。
- **操作ボタン**: 再生・判定/選別への導線・いいね・あとで見る解除。
- **バッジ**: 該当Tier・あとで見る・利用不可。状態は `ステータス: Tier1 未判定 / Tier1 判定済み / Tier2 未選別 / Tier2 選別済み` の文言で表す。
- **Tooltip に逃がす説明**: **解除ルールはタイトル横「あとで見る」Tooltip に集約**（判定/選別完了 or 処理済み動画へのいいねで自動解除／通常再生・AVP候補追加では解除されない／AVP再生でも自動解除しない）。カード内注記は増やさない。
- **空状態**: セクションごとに「該当なし」。
- **利用不可状態**: バッジ＋薄表示。
- **あとで見るとの関係**: 本画面の主対象（DB 相当）。
- **AVP候補との関係**: **混同しない**（別概念・別保存先）。AVP再生ではあとで見るを自動解除しない前提で見せる。
- **処理済み候補**: **個別解除を基本**とする。一括解除を残すかは **未決**（§16 確認項目）。
- **レイアウト**: PC 幅では **5列** を想定。
- **他画面と共通化すべき部品**: 共通カード。

### AVP
- **目的**: 候補から最大4本を選び並列再生する。
- **主な表示領域**: **上段=AVP候補テーブル**、**下段=2×2 再生セット**。
- **カード優先/テーブル優先**: 上段テーブル（ランキング/検索と操作付きテーブル土台を共有）、下段カード（Tier1ライブラリの動画カードと整合）。
- **主役にする情報（上段列）**: タイトル（truncate、全名は Tooltip）／**総合スコア**（`123.4 pt（総合 12位）`、既定=再生可能だけの順位、全動画へ切替可）／**総合順位**／**視聴日数**（APP_PLAYBACK 基準）／**いいね**（上段は表示のみ・クリック不可）／**あとで見る列**（バッジでなく列）／**Tier1 列**（未判定/Lv0..Lv4/—）／**Tier2 列**（未選別/Lv0..Lv4/—）。
- **出さない情報**: スロット番号（1/2/3/4 は出さず順序のみ）・視聴回数。
- **操作ボタン**: 上段=再生対象に追加 / 候補から外す。下段=再生 / 再生対象から外す / **個別いいね** / **一括いいね**（未いいねのみ ON、既いいねは解除しない。ボタン名候補「4本をまとめていいね」）／**全候補をクリア** ／ **再生対象をクリア**。`AVPで再生` ボタンは2×2の右上固定。
- **バッジ**: テーブルはバッジを避ける（利用不可=行薄表示＋追加不可、再生中=行ハイライト）。下段カードは再生中=カードハイライト（「視聴済み」ではなく「現在/直近で再生中」）。
- **Tooltip に逃がす説明**: タイトル横「AVP」Tooltip に説明集約。
- **空状態**: 候補なし/再生対象なしの別表示。
- **利用不可状態**: 行薄表示＋追加無効。
- **あとで見る/AVP候補との関係**: 候補=localStorage 相当、あとで見る=DB 相当として混同しない。**AVP再生であとで見るを自動解除しない**前提で見せる。
- **他画面と共通化すべき部品**: 上段はランキング/検索のテーブル土台、下段は Tier1 ライブラリカード。
- **不変**: 候補上限なし・再生対象最大4本・欠落ID掃除・再生後クリア。

- **カード優先/テーブル優先**: テーブル優先。
- **主役にする情報**: 総合スコア・視聴日数・いいね。
- **通常列**: 順位 / タイトル / 総合スコア / 視聴日数 / いいね / Tier1 / Tier2 / 操作。

### 検索

## 14. 暫定的に決めるデザイン項目（最終決定ではない）

| 項目 | 暫定方針 | 理由 |
|---|---|---|
| 該当Tierバッジの色 | 寒色モダンの範囲で Tier1=青系 / Tier2=シアン or ティール系（暫定） | GLOSSARY が「具体色は統合 Variant K で全体テーマに合わせて決める」と明示。寒色内で2色を分けると整合する。最終色は確認項目。 |
| Tier1 / Tier2 の見分け方 | 色相差＋ラベル文字（Tier1/Tier2）で二重符号化 | 色だけだと色覚差で識別しづらいため、ラベル併記で冗長化。 |
| あとで見るバッジの見た目 | 中立トーンの控えめなアウトラインバッジ | DB 状態だが操作の主役ではないため、該当Tierより弱い視覚重み。 |
| 利用不可バッジの見た目 | 低彩度＋全体薄表示（テーブルは行薄表示） | 「操作不可」を即伝えるため、彩度を落とし他バッジより沈める。 |
| AVP候補バッジを初期ONにするか | **初期 OFF**（既定） | フィードバックで「初期 ON にしない」と記録済み。初期 ON バッジを増やすと密度が上がるため。覆す場合は確認項目。 |
| 再生中ハイライト | amber 系（暫定） | バッジでなくハイライトで「今/直近再生中」を示す共通ルール。寒色テーマの中で暖色は注意を引きやすく、状態の一時性を表現できる。 |
| カード密度 | 中密度（2行タイトル＋メタ3項目＋バッジ＋操作） | 情報過多を避けつつ視聴日数/作成日/該当Tierを主役にできる下限。 |
| テーブル密度 | 高密度（行高さ詰め・数値右寄せ） | ランキング/検索/AVP候補は数値比較が目的のため、一覧性を優先。 |
| 余白 | シェル外周は広め、カード/テーブル内は詰め | 統合シェルの落ち着き（寒色モダン）と情報密度の両立。 |
| ボタンの優先度 | 主操作（再生/引く/スキャン）= primary、状態操作（いいね/あとで見る/AVP候補）= secondary/ghost | 主操作を一目で識別できるよう優先度を段階化。 |
| Tooltip に逃がす説明の範囲 | タブの意味・解除ルール・スコア内訳・条件説明など「長文の補足」のみ | 主要ボタンのラベルが自明なものには付けず、説明過多を防ぐ。 |
| 寒色モダンテーマの使い方 | 背景/サイドバーは寒色低彩度、強調は primary 青系、状態色のみ例外（amber=再生中、利用不可=低彩度） | Variant J 系の方向性を全画面に統一しつつ、状態色だけ意味付きで例外運用。 |
