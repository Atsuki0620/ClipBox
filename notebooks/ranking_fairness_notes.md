# 総合ランキング公平化 分析メモ

## 目的

`notebooks/ranking_fairness_analysis.ipynb` は、総合ランキングを変更する前に現行式の偏りや取りこぼし候補を確認するための分析成果物です。アプリ本体、UI、API、DB スキーマ、ランキングロジックは変更していません。

## 使用データ

- ローカルの `data/videos.db`
- 対象テーブル: `videos`, `viewing_history`, `judgment_history`, `likes`
- DB は `sqlite3` の read-only URI `mode=ro` で開き、`PRAGMA query_only=ON` を設定します。
- DB が存在しない場合はダミーデータを作らず、想定パスを表示して分析をスキップします。

## 確認観点

- 現行の総合スコア式で Top 50 がどの要素に寄っているか。
- `総合Top50`, `非Top50`, `score=0` の間で、再生済み率、いいね率、判定済み率、Tier2選別済み率に差があるか。
- 非Top50の中に、いいね、Lv3以上、複数日視聴、選別済みなどの良い兆候がある作品が残っていないか。
- 最終再生日なし、または90日以上未再生の作品に発掘余地がないか。

## 発掘候補の仮説

- A: 非Top50で、スコアは比較的高いが再生回数または判定回数が少ない候補。
- B: Lv判定済みだがTier2未選別の候補。
- C: 非Top50で、いいね、Lv3以上、複数日視聴、選別済みのいずれかがある候補。
- D: 未再生または90日以上前の再生で、かつ良い兆候がある候補。
- E: `discovery_score` の試算。これは仕様ではなく、比較用の分析仮説です。

## 仕様化前の論点

- 総合ランキング自体を変えるのか、別枠の発掘候補として追加するのか。
- 発掘候補を UI に出す場合、候補数、並び順、期間条件、利用可能フィルタをどう扱うか。
- `discovery_score` を採用する場合、係数と説明責任をどう持つか。
- 個人パスや巨大な候補一覧はドキュメントに残さず、notebook 再実行時に確認する。
